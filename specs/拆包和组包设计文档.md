# 可靠 UDP 分片传输（拆包 / 组包）设计文档 · 最小可落地版（MVP）

> 适用范围：`examples/sync-demo/udp_transport.{h,cpp}`（sync-demo 与 sync-suite 共用同一份传输层实现）
> 关联需求：生产环境带宽很小 → 每个 UDP 数据报 ≤ 800 字节 → 强制多分片 → 必须补齐分片重组的健壮性
> 本版原则：**最小可落地**。只做「全量重传 ARQ」——用最少机制修复健壮性缺口并跑通 800B。选择性重传（NACK）、发送节流、组件大拆分等均**不在本版范围**。
> 已经过 **两轮 Codex 评审** 并按其 critical/high 意见收敛（见文末附录）。

---

## 1. 背景与目标

### 1.1 为什么要做

当前 `UdpFileTransport` 的分片阈值 `kFragSize = 60000`，demo 工件（同步 changeset、约 1702B 的快照）几乎都小于它，运行时几乎恒为「单片直通」——分片/重组代码路径基本休眠。生产要求每个 UDP 数据报 ≤ **800B**，一旦压到 800B，任何工件都会被切成多片，随之暴露三个此前「存在但从未被触发」的健壮性缺口：

1. **无重传 / 无 ACK**：任一分片丢失 → 接收端 `complete()` 永不成立 → 半成品 `PendingMsg` 永久滞留（内存泄漏）→ 文件永远到不了 inbox → 上层卡到超时。片越小、片越多，多分片消息整体丢失概率越高。
2. **无 PendingMsg 超时 / 淘汰**：滞留半成品不被清理。
3. **msg_id 为随机 32bit**：两个并发文件的 msg_id 碰撞会把分片混进同一 `PendingMsg` 致数据损坏。

### 1.2 MVP 目标

用**最少的机制**把「拆包 + 组包」做到健壮且可测：

- **对外接口**：设置「每次传输最大字节数」（数据报总长上限）；
- **拆包**：文件 → N 个 ≤ maxBytes 的自描述数据报；
- **组包**：乱序 / 重复 / 丢失 / 越界 / 元数据不一致 / 恶意文件名下正确重组，并**逐条修复三缺口**：
  - 丢包恢复 = **全量重传 ARQ**（发送端超时未收 ACK 就整条重发，直到 ACK 或放弃）；
  - 泄漏 = 接收端 pending **超时淘汰** + 发送端超 `maxRetries` **释放**；
  - 碰撞 = 归并键含**发送端点** + msg_id 改**单调计数器（随机起点）**。
- **验证**：sync-demo 与 sync-suite 把每次传输最大字节数设为 **800B**，端到端跑通。

## 2. 现状分析

### 2.1 现有拆包 / 组包

| 环节 | 现状（`udp_transport.cpp`） | 问题 |
|------|------|------|
| 拆包 | `sendFile()`：`readAll` → 按 `kFragSize` 切片 → 每片加 17B 头逐个 `writeDatagram` | 阈值是「净荷」非「数据报总长」；发完即不管，无重传 |
| 组包 | `handleDatagram()` + `PendingMsg`：按 `msg_id` 归并，`complete()` 判齐，`assemble()` 顺序拼接 | 无超时淘汰；键只用 `msg_id`；丢片即永久卡死；**不校验 `frag_idx` 越界与元数据一致性**；**用网络包里的 filename 直接拼 inbox 路径** |
| 归并键 | `QMap<quint32 msgId, PendingMsg>` | 跨发送端 `msgId` 可碰撞 |
| 出站去重 | `sentFiles_`（内存）+ 发完改名 `.sent` | 内存态与磁盘态双份、重启不兜底 |
| 并发 | `run()` 单线程内「收 → 发」交替；`stop_` 原子 | 单线程 → ARQ 状态可**无锁**托管在 `run()` 内 |

现有 DATA 头 = 17 字节：`magic(4) | msg_id(4) | frag_idx(2) | frag_count(2) | total_size(4) | fname_len(1)`。

### 2.2 决定设计形态的三个关键约束

- **`run()` 单线程收发交替**：接收与发送在同一线程顺序执行 → **所有 ARQ 状态可为线程私有、无需加锁**；且「组齐 → durable 落盘 → 记入短表」在同一线程内**顺序完成**，其间不会插入其它数据报（无并发交付窗口）。
- **配置须在 `start()` 前设置**：`maxTransmitBytes` 等设定后 `run()` 只读 → 无数据竞争。
- **同一 socket 收发**：socket 绑定 `localPort_` 并用它发送，故其数据报源端口即 `localPort_`；接收端得到的 `(sender, senderPort)` 就是发送端监听端点，**回发 ACK 天然可达**。

`examples/sync-demo/main.cpp`（中心1+边缘3）、`examples/sync-suite/Scenario1Runner.cpp`（中心1+边缘3）、`examples/sync-suite/Scenario2Model.cpp`（中心A/子B 各1）。

## 3. 设计原则（MVP）

1. **「最大字节数」= 数据报总长上限，不是净荷。** 净荷 = maxBytes − 头长 − 文件名长，且**逐文件计算**（文件名嵌在每片、长度可变）。简单把 `kFragSize` 设成 800 会得到 `≈850+` 的数据报，反而违约。
2. **先做全量重传 ARQ，不做 NACK。** 生产工件仅 KB 级，全量重传（超时未收 ACK 就整条重发）足够健壮且实现最简；选择性重传是后续优化。
3. **RTO 兜底永不被关闭。** 只要一条消息未收到 ACK，发送端就按 RTO 周期性重发，直到 ACK 或 `maxRetries`——不引入任何「收到某种反馈后停止重传」的开关（那会在反馈丢失时永久卡死）。
4. **入站严格校验，宁丢勿污。** 对每个 DATA 片校验类型/大小/越界/元数据一致性/组装总长/**文件名安全**；任一不过即丢弃该片或淘汰该消息，绝不静默交付截断/损坏/越界的数据。
5. **交付以 durable 落盘为准，ACK 只在落盘成功后发。** 「组齐（reassembled）」≠「已交付（durable）」：接收端组齐后先尝试写 inbox（payload + `.ready`），**两步都成功**才记入「已完成短表」并回 ACK；写失败则不 ACK，交由发送端 RTO 重传自愈。
6. **无锁 + 单调时钟。** ARQ 状态托管在 `run()` 单线程；用 `QElapsedTimer` 取单调毫秒作所有超时基准；计数器/表均普通类型，无需 atomic。
7. **归并键含发送端点。** 接收端 `pending`/`completed` 键 = `(senderKey, msgId)`；发送端 `msgId` 改每 transport 单调计数器、**随机起点**（缓解「重启后从 1 重来」与旧包碰撞）→ 碰撞在设计上根除。
8. **出站状态以磁盘后缀为唯一真源。** 用**原子认领**（单次 rename `.ready→.ready.sending`）替代内存 `sentFiles_`；每次 rename 检查返回值。

## 4. 设计方案（MVP）

### 4.1 分片与「最大字节数」

- 新增「数据报类型」字段后，**DATA 固定头 = 18B**：`magic(4) | type(1) | msg_id(4) | frag_idx(2) | frag_count(2) | total_size(4) | fname_len(1)`。
- 每片净荷 `M = maxTransmitBytes − 18 − fname_len`。
- 分片阶段守卫（任一不过 → `FragmentResult{ok=false}`，文件改 `.failed`，不进 outbound）：
  - `data.size() > UINT32_MAX`（`total_size` 为 `quint32`）→ 失败；
  - 文件名 UTF-8 > 255（`fname_len` 为 `quint8`）→ 失败；
  - `fragCount64 = data.isEmpty() ? 1 : ceil(total / M)`；`fragCount64 > 65535`（`frag_count` 为 `quint16`）→ 失败。
  - 空文件特判 1 片（空体也要被传输并触发落盘）。
- **下限拒绝而非上夹**：`maxTransmitBytes` 若 `< 274`（`= 18 头 + 255 最长合法文件名 + 1 净荷`）则**拒绝设置**（保持默认、返回 false），不静默突破调用者给的上限。上限 65507。800 远高于 274，安全。
  - 例：文件名 `snapresp__<12>.payload`（约 30B），800B 下净荷 `M = 800 − 18 − 30 = 752`，1702B 快照 → `ceil(1702/752) = 3` 片。

### 4.2 线格式（大端；`type` 紧跟 `magic`，收端先校验 `magic + type` 再分派）

| 类型 | type | 布局 | 长度约束 |
|------|:----:|------|------|
| **DATA** | `0x01` | `magic \| 0x01 \| msg_id \| frag_idx \| frag_count \| total_size \| fname_len \| fname[N] \| payload[M]` | `≥ 18 + fname_len` |
| **ACK**  | `0x02` | `magic \| 0x02 \| msg_id` | **精确 9 字节** |

- DATA 各分片的 `msg_id / frag_count / total_size / fname_len / fname` 完全相同，`frag_idx / payload` 各异——故接收端**不论哪片先到都能初始化元数据**。
- **ACK 校验**：`dg.size() == 9 && magic 正确 && type == 0x02`，否则丢弃（防截断/伪造/带尾字节的控制包）。
- ACK 由接收端在「消息组齐**且 durable 落盘成功**」后回发给该消息的发送端点。

### 4.3 MVP ARQ 协议（全量重传）

**所有消息（含空文件、单片）走同一路径**——单片消息也进 outbound、也要等 ACK 或 giveUp，避免语义分叉。

**发送端** —— `run()` 内的 `QHash<quint32 msgId, Outbound>`；`Outbound { destHost, destPort, msgId, datagrams[], baseName, lastSentAt, retries }`：
- **首发**（严格「先落定 `.sending` 终态对象、再发送」的顺序，杜绝孤儿/终态错位）：
  1. **原子认领**：`rename(name.ready → name.ready.sending)`；失败=被并发认领/竞态 → 跳过。
  2. **改 payload 为 `.sending`**：`rename(name.payload → name.payload.sending)`；失败 → `rename(ready.sending → .failed)` + 告警 → 跳过（罕见 FS 错误）。此步在**发送之前**，保证之后 ACK/giveUp 的终态改名有确定对象。
  3. **读 payload**（`Result`）：读不到/打不开/读失败 → 两文件改 `.failed` + 告警、**不进 outbound**（不误发空/残缺数据）。
  4. **拆包** `fragmentMessage`：失败（§4.1 守卫）→ 两文件改 `.failed` + 告警、**不进 outbound**。
  5. 全部分片发出 → 记入 outbound（`lastSentAt = now, retries = 0`）。
  所有 rename/读取都检查返回值；1–4 任一失败都**不进 outbound**（与 §5.1/§5.2 一致）。
- **RTO 兜底**：`now − lastSentAt > RTO` 时——若 `retries < maxRetries`：**全量重发**所有分片（从 outbound 内存中的 datagrams，不再读盘）、`retries++`、刷新 `lastSentAt`；否则：放弃，文件改名 `.failed` + `qWarning`（`E_UDP_DELIVERY_GIVEUP` 语义），删除 outbound。
- **收 ACK**：先过 §4.2 ACK 校验；再校验 ACK 的 UDP 来源端点 == `outbound[msgId].dest`（不匹配则丢弃，防误包/旧包）；匹配则文件改名 `.sent`、删除 outbound。

**接收端** —— `FragmentReassembler`，`pending` 键 = `(senderKey, msgId)`，`PendingMsg { filename, totalSize, fragCount, frags{idx→payload}, createdAt, lastProgressAt }`；另有「已完成短表」`completed{(senderKey,msgId) → completedAt}`：
- **`feed(senderKey, dg, now)`**：先过 §4.4 校验；
  - 若 `(senderKey,msgId)` 在「已完成短表」→ ACK 丢失导致的重发 → 返回 **`NeedAck`**（**不重建 pending、不落盘**）；
  - 新消息 → 建 pending（`createdAt = lastProgressAt = now`）；已存在 → 校验元数据一致；
  - 存片：仅当 `frag_idx` 之前未收到过才刷新 `lastProgressAt`（**重复片不刷新进度**）；
  - `frags.size() == fragCount`（`complete()`）→ `assemble()` 并校验 `size == totalSize`：通过则**从 pending 移除**、返回 **`Completed(filename, bytes)`（此时尚未记入短表）**；不通过则丢弃 pending + 告警（损坏）。
- **`markDelivered(senderKey, msgId, now)`**：由外层在「payload + `.ready` 均写入成功」后调用，才把该键记入「已完成短表」。
- **`evictStale(now)`**：`now − lastProgressAt > reassemblyTimeout` 的 pending → 淘汰释放内存；`now − completedAt > completedRetention` 的短表项 → 清理。

**外层 `run()` 的交付闭环（关键）**：`feed` 返回 `Completed` → 尝试写 inbox（先 payload、后 `.ready`）→ **两步都成功**：`markDelivered` + 回 ACK；**任一失败**：不回 ACK + 告警（发送端 RTO 会重传；因该 msg 既不在 pending 也不在短表，重传分片会重建 pending → 再组齐 → 再尝试落盘）。因 `run()` 单线程串行，`Completed→写盘→markDelivered` 之间不会处理其它数据报，**无重复交付窗口**。

**典型时序（丢第 k 片）**
```
子B: DATA[0..n-1] ──(第 k 片丢)──► 中心A: 缺 idx=k，不 complete
子B: (RTO 到、未收 ACK) 全量重发 DATA[0..n-1] ──► 中心A: 收齐 → 校验 → 写 inbox 成功 → markDelivered → 回 ACK
中心A: ACK(msg) ────────────────────────────────► 子B: 改 .sent、删 outbound
```

### 4.4 入站严格校验规则（组包健壮性核心）

对每个 DATA 片依次校验，任一不过即丢弃该片（元数据类不一致则可淘汰整条 pending）：
1. `dg.size() ≥ 18`；`magic` 正确；`type == 0x01`。
2. `fragCount > 0`；`fragIdx < fragCount`（**拒绝越界片**——否则越界片可把 `frags.size()` 凑满 `fragCount` 致 `assemble()` 静默截断）。
3. `dg.size() ≥ 18 + fname_len`（文件名长度不超实际报文）。
4. **文件名安全（防路径穿越）**：`filename` 必须是安全 basename——非空、UTF-8 有效、不含 `/` 与 `\`、非绝对路径、不含 `..` 路径段；否则丢弃（否则以后若不止 bind loopback，`../`/绝对路径可越界写文件）。
5. 该 `(senderKey,msgId)` 已有 pending 时：本片 `fragCount / totalSize / fname_len / fname` 必须与首片**逐字段一致**，否则丢弃该片（严重时淘汰 pending + 告警）。
6. 组齐后：`assemble().size() == totalSize`，否则判损坏、丢弃 pending。

### 4.5 三个缺口 → 对应处理方案（MVP）

| 缺口 | 处理方案 |
|------|---------|
| **① 无重传 / 丢片卡死** | **全量重传 ARQ**：发送端按 RTO 周期性整条重发，直到收 ACK 或 `maxRetries`；RTO **永不被任何反馈关闭**；放弃后改 `.failed` + 告警，上层可感知。交付以 durable 落盘为准（ACK 只在落盘成功后发） |
| **② 无淘汰 / 内存泄漏** | 接收端 pending 记 `lastProgressAt`，`run()` 每轮 `evictStale()` 淘汰超 `reassemblyTimeout` 无进展者；发送端超 `maxRetries` 释放 outbound；「已完成短表」按 `completedRetention` 过期。三处共同保证内存有界 |
| **③ msg_id 碰撞** | 归并键改 `(senderHost, senderPort, msgId)`；`msgId` 改每 transport 单调计数器、随机起点。两并发文件即便同 msgId 也因发送端点不同而隔离 |

### 4.6 参数与时间约束（保守，含抖动预算）

**记号**（相对「首发时刻」）：
```
t_lastData ≈ maxRetries × RTO          // 发送端最后一次 DATA 发出的大致时刻
t_giveUp   ≈ (maxRetries + 1) × RTO     // 再过一个 RTO 仍无 ACK → 放弃并删除 outbound
J（抖动预算）= pollMs + 一条消息全部分片的发送耗时 + run() 被其它 outbox 文件占用的时间  // 经验值，保守取大
```
**约束**（必须满足，否则会误淘汰 / 重复落盘）：
```
pollMs ≤ RTO
completedRetention ≥ t_giveUp + J   // 保证「giveUp 前最后一次重传」仍命中短表 → 只再 ACK、不重复落盘
reassemblyTimeout  ≥ t_giveUp + J   // 发送端仍在重传时不淘汰接收端 pending
```
**取舍**：即便 pending 被提前淘汰，后续任一次「全量重传」会作为**新消息**安全重建 pending → 可能牺牲个别恢复率，但**绝不破坏幂等/正确性**（重传的整条会重新组齐、重新落盘；若已 markDelivered 则命中短表只回 ACK）。

**默认值**（demo **保守**取值，已含 J 预算；非严格证明的最小值）：

| 参数 | 默认 | 说明 |
|------|------|------|
| `maxTransmitBytes` | 60000（**测试设 800**） | 单个 UDP 数据报总长上限 |
| `pollMs` | 50 ms | 收发轮询间隔（不变） |
| `RTO` | 500 ms | 无 ACK 时的整条重传间隔 |
| `maxRetries` | 5 | 放弃前重传上限（→ `t_giveUp ≈ 3000 ms`） |
| `reassemblyTimeout` | 8000 ms | 接收端半成品无进展淘汰阈值（≥ `t_giveUp + J`） |
| `completedRetention` | 8000 ms | 已完成短表记忆时长（≥ `t_giveUp + J`） |

> 定时器基于 `waitForReadyRead(pollMs)` 轮询，属**软定时器**：实际触发有 `pollMs + 发送耗时` 级抖动，故上表默认值已把该抖动计入 `J`；测试断言亦需容忍该抖动。

### 4.7 对外接口（`udp_transport.h`）

## 5. 设计实现（MVP）

### 5.1 组件（均在 `udp_transport.{h,cpp}` 内）

**① 拆包（纯函数，可单测）**
```cpp
struct FragmentResult { QList<QByteArray> datagrams; bool ok = false; QString error; };
// 把一条消息切成若干 ≤ maxTransmitBytes 的 DATA 数据报。
// 空 data → 1 个空净荷片(ok)。data>UINT32_MAX / 文件名 UTF-8>255 / fragCount>65535 → ok=false + error。
static FragmentResult fragmentMessage(quint32 msgId, const QString& filename,
                                      const QByteArray& data, int maxTransmitBytes);
```

**② 组包（有状态机，可脱离 socket 单测——组包健壮性的核心测试对象）** `class FragmentReassembler`
```cpp
struct RxEvent {
    enum Kind { None, Completed, NeedAck } kind = None; // Completed:已组齐待外层落盘; NeedAck:重复消息只回ACK
    quint32 msgId = 0;
    QString filename;   // Completed 时有效
    QByteArray bytes;   // Completed 时有效（还原出的整文件）
};
explicit FragmentReassembler(int reassemblyTimeoutMs, int completedRetentionMs); // 单测可注入短超时
RxEvent feed(const QString& senderKey, const QByteArray& datagram, qint64 nowMs); // 校验+存片；组齐→Completed(不进短表)
void markDelivered(const QString& senderKey, quint32 msgId, qint64 nowMs);        // 外层 durable 落盘成功后调用→入短表
void evictStale(qint64 nowMs);   // 淘汰超时 pending + 过期短表
int  pendingCount() const;       // 供测试断言「无泄漏」
```
§4.4 全部校验、乱序 map 存片、`complete()`、`assemble()` + 总长校验、去重、短表，都在此类内。**注意** `feed` 返回 `Completed` 时不写短表；只有外层落盘成功后调 `markDelivered` 才写——这样落盘失败不会被误判为「已交付」。

**③ 发送在途**：MVP **不单独拆类**，直接在 `run()` 内用 `QHash<quint32, Outbound>` 承载；`msgSeq_` 为普通 `quint32` 成员、构造时随机初始化（`QRandomGenerator`），单线程内 `++msgSeq_`（无需 atomic）。

**④ `QThread::run()` 编排**（保持既有「收 → 发」骨架，插入维护步）
```
bind(localPort_); mkpath(inbox); QElapsedTimer clk; clk.start();
FragmentReassembler reasm(reassemblyTimeoutMs_, completedRetentionMs_); QHash<quint32,Outbound> outbound;
while (!stop_.loadAcquire()):
    if sock.waitForReadyRead(pollMs):
        while sock.hasPendingDatagrams():
            读 dg + (sender,port); key = endpointKey(sender,port); now = clk.elapsed()
            switch parseType(dg):   // 先校验 magic
                DATA: ev = reasm.feed(key, dg, now)
                      if ev.kind==Completed:
                          if writeInbox(ev.filename, ev.bytes) && writeReady(ev.filename):
                              reasm.markDelivered(key, ev.msgId, now); sendAck(ev.msgId)→sender
                          else: qWarning("写 inbox 失败"); /* 不 ACK，发送端会重传 */
                      if ev.kind==NeedAck: sendAck(ev.msgId)→sender
                ACK : if dg.size()==9 && outbound.contains(msgId) && outbound[msgId].dest==(sender,port):
                          rename(base.payload.sending→.sent); outbound.remove(msgId)
    pollOutbox(sock, outbound, clk):
        对每个 *.ready 就绪工件:
            if !rename(name.ready → name.ready.sending): continue          // ① 原子认领；失败=被抢/竞态，跳过
            if !rename(name.payload → name.payload.sending):               // ② 发送前先落定 .sending 终态对象
                rename(name.ready.sending → .failed); qWarning; continue   //    罕见 FS 错误 → 终态 .failed，跳过
            Result<QByteArray> pr = readPayload(name.payload.sending)      // ③ 读取用 Result
            if !pr.ok:
                rename(payload.sending→.failed); rename(ready.sending→.failed); qWarning; continue  // 不发送、不进 outbound
            r = fragmentMessage(++msgSeq_, name, pr.data, maxTransmitBytes_)   // ④ 拆包
            if !r.ok:
                rename(payload.sending→.failed); rename(ready.sending→.failed); qWarning; continue  // 不发送、不进 outbound
            逐片发往 dest(单/多 peer 路由不变，见 extractTargetPeer)          // ⑤ 发送 → 才入 outbound
            outbound.insert(msgId, {dest, msgId, r.datagrams, base, lastSentAt=now, retries=0})
    for e in outbound:   // RTO 兜底
        if now - e.lastSentAt > RTO:
            if e.retries < maxRetries: 全量重发 e.datagrams→e.dest; e.retries++; e.lastSentAt=now
            else: rename(base.*→.failed); qWarning; outbound.remove(e.msgId)
    reasm.evictStale(now)
```
- `pollOutbox` 的「`.ready` 就绪协议」「单/多 peer 目标解析 `extractTargetPeer`」**保持不变**；仅把「切片 + `writeDatagram`」替换为 `fragmentMessage`，把「发完改 `.sent`」改为「原子认领 + `.sending` + 由 ACK/giveUp 决定终态」。
- `endpointKey(host,port)` = `host.toString()+":"+port`，仅作 map 查找键；发送端点的结构化信息（`QHostAddress`+port）另存于 `Outbound.dest`，**从不反解析该字符串**，故 IPv6 字符串形态不影响正确性。

### 5.2 交付状态机（磁盘后缀为唯一出站真源，取代 `sentFiles_`）

```
{name}.payload + {name}.payload.ready
   │  pollOutbox 扫到 *.ready
   ▼  ① 原子认领：rename(.ready → .ready.sending)   ← 单次 rename 即离开扫描集，天然防并发重认领/重发
{name}.payload + {name}.payload.ready.sending
   ▼  ② 发送前：rename(payload → payload.sending)   ← 认领后立即改名，使后续终态改名有确定对象
{name}.payload.sending + {name}.payload.ready.sending
   │  ③读 payload / ④拆包 任一失败 → rename(两者 → .failed) + 告警，【不进 outbound】
   │  ⑤ 否则发送全部分片 → 记入 outbound
   ├─ 收 ACK  → rename(→ .sent)     ，删 outbound
   └─ giveUp → rename(→ .failed)+告警，删 outbound
```
- **删除内存 `sentFiles_`**：认领即由 `.ready→.ready.sending` 的一次 rename 完成，磁盘后缀是唯一出站状态源，消除「内存态/磁盘态不一致」的理解成本。
- 每次 rename 检查返回值，失败即 `qWarning`；认领失败者直接跳过、不进 outbound。
- **崩溃恢复不在 MVP 范围**：进程崩溃遗留的 `.sending` 孤儿不做重投递（两个 demo 每次 `setup` 都清空 workspace，孤儿无影响）。

## 6. 设计测试（MVP）

### 6.1 单元测试（确定性、注入丢包；新增 `tests/unit/tst_udp_reassembly.{cpp,pro}`）

纯组件测试，**不依赖真实 socket / 网络**——直接调 `fragmentMessage` 产出数据报、**按需扣留若干**后喂给 `FragmentReassembler`，断言行为。丢包即「不喂某些片」，**对生产代码零侵入**（无需任何生产丢包钩子）。淘汰/短表用例通过 `FragmentReassembler(小超时)` 注入短超时来快速触发。

| 用例 | 场景 | 断言 |
|------|------|------|
| `frag_sizes` | data size ∈ {0,1,M,M+1,多片} × 文件名长度 | 分片数正确；每报 ≤ maxBytes；头字段正确 |
| `roundtrip_inorder` | 顺序喂全部片 | `Completed`，还原 bytes 与原文逐字节一致 |
| `roundtrip_reorder` | 乱序 + 重复喂 | `Completed` 且一致（乱序/重复幂等，重复不刷进度） |
| `retransmit_recovery` | 先扣留第 k 片 → 不 complete → 再把整条重喂 | 补喂后 `Completed`（模拟全量重传恢复） |
| `durable_delivery` | `Completed` 后**不调** `markDelivered`（模拟落盘失败）→ 再喂整条 | 再次 `Completed`（可重新交付）；调 `markDelivered` 后再喂 → `NeedAck` |
| `eviction` | 半成品静置超 `reassemblyTimeout` → `evictStale` | `pendingCount()==0`（无泄漏） |
| `collision` | 两个不同 `senderKey` 用相同 `msgId` 交替喂 | 各自独立 `Completed`，内容不串扰 |
| `dup_after_complete` | `markDelivered` 后再喂同消息 DATA | 不重复交付；返回 `NeedAck` |
| `reject_out_of_range` | `frag_idx >= frag_count` | 被丢弃；不污染 pending、不误判 complete |
| `reject_meta_mismatch` | 后续片 `fragCount/totalSize/fname` 与首片不一致 | 被丢弃 |
| `reject_size_mismatch` | 组齐但 `assemble().size() != totalSize` | 不交付、判损坏 |
| `reject_unsafe_filename` | filename 含 `..` / `/` / 绝对路径 | 被丢弃 |
| `frag_errors` | data>UINT32_MAX（逻辑分支）/ 文件名>255 / `fragCount>65535` | `fragmentMessage` 返回 `ok=false + error` |

## 8. 风险 · 取舍（MVP）

- **线格式不兼容**：新增 `type` 字段与旧 17B 头不兼容。因 sync-demo / sync-suite 同源、同构建、一体升级，无跨版本包袱（运行中新旧二进制混用会互不识别，demo 场景不存在）。
- **默认 60000 行为微变**：旧 `kFragSize=60000` 是「净荷」，新 60000 是「数据报总长」，分片边界略有不同；但 demo 工件均 < 60000、实际仍单片，**无可观测差异**（如实说明，不宣称「完全不变」）。
- **崩溃恢复**：MVP 不处理进程崩溃遗留的 `.sending` 孤儿（demo 每次 setup 清 workspace）。生产化需要「重启扫描 `.sending` 重投或清理」的补充设计。
- **全量重传的带宽代价**：高丢包时整条重发比选择性重传费带宽；但 demo 工件 KB 级、loopback 近无损，代价可接受。选择性重传（NACK）留作后续优化。
- **背靠背发包**：800B 下大工件分片数增多，`run()` 背靠背 `writeDatagram` 可能压满内核发送缓冲；本版不做发送节流——依赖「送达以 ACK 为准 + RTO 全量重传」自愈（发送失败/被丢的片会随下一次 RTO 重发）。发送节流留作后续优化。

## 10. 术语澄清（避免与「拆包组包」混淆）

- **`.ready` 两步就绪协议**（先写 `.payload` 再写 `.payload.ready`）：是「文件写完了 / 可安全读了」的**完成信号**，不是分片拆组。
- **`encodeSnapshots` / `PayloadCodec`**：是**序列化 / 编解码**（结构 ↔ 一个字节块），产出/消费「一个整块」，整块之后才交给传输层分片——不是网络分片。
- 真正的网络「拆包 / 组包」只在 `UdpFileTransport` 一处：本设计的 `fragmentMessage`（拆）与 `FragmentReassembler`（组）。

## 附录：对两轮评审意见的取舍（最小可落地）

**第一轮已纳入**：数据报总长上限、拒绝下限而非上夹、`type` 字段、`fragmentMessage` 错误结果、`fragCount>65535` 守卫、`(端点,msgId)` 归并 + 单调 msgId、入站严格校验（越界/元数据/总长）、RTO 永不被反馈关闭的全量重传、`maxRetries` + 交付状态机、ACK 绑定来源端点、pending 超时淘汰、已完成短表、空/单片同路径、重复片不刷新进度。

**第二轮已纳入**：
- 交付幂等：`feed` 返回 `Completed` 但不进短表，外层 durable 落盘成功后 `markDelivered` 才入短表 + 回 ACK（落盘失败不误判已交付）。
- 出站状态机收紧：原子认领（`.ready→.ready.sending`）+ 逐次 rename 返回值检查 + **删除 `sentFiles_`**（磁盘后缀为唯一真源）；崩溃孤儿明确不在范围。
- 入站文件名安全（防路径穿越）；ACK 精确 9 字节校验；`total_size>UINT32_MAX` 守卫。
- 时间约束改为 `t_giveUp = (maxRetries+1)×RTO` 并含抖动预算 `J`，保守默认抬到 `8000/8000`，诚实标注「保守值非严格证明」。
- **删除** `setReliabilityParams` 公开接口（超时改编译期常量；单测经 `FragmentReassembler` 构造注入）；`msgSeq` 用普通 `quint32` + 随机起点（非 atomic）。

**明确不在本版**（后续增量）：NACK 选择性重传及其多包/容量派生、`OutboundBook` 独立拆类、发送节流 / datagram 预算、生产内置丢包注入钩子（单测以「扣留分片」替代）、会话 nonce 入线格式（以随机 msgSeq 起点替代）、`(dest,msgId)` 复合 outbound 键（仅未来「一条消息广播多 peer」才需要）、崩溃后 `.sending` 孤儿恢复。
