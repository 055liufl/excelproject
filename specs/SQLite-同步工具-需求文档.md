## 0. 决策基线（评审基线）

| 决策项 | 选定方案 | 理由 |
| --- | --- | --- |
| 传输层 | 文件制品 + outbox/inbox 目录监听，传输通道作为与同步逻辑无关的黑盒第三方工具 | 解耦同步逻辑与传输实现，适配 2Mbps 低带宽及不可控网络环境，便于离线、断点续传与审计 |
| 变更捕获 | SQLite Session Extension 生成二进制变更集（changeset） | 原生增量捕获、变更集紧凑，契合低带宽；代价为需重编译启用扩展，并要求句柄穿透 sqlite3* |
| 场景 2 合并 | 工具提供差异对比 / 合并 API，由宿主应用自建 GUI | 比对与合并策略多样且业务相关，工具只提供能力，界面与人工裁决交还宿主，避免绑定特定 UI |
| 域拓扑 | 单域星型（中心 A + 子节点 B/C/D），预留多域扩展 | 当前场景为单中心星型，先满足主路径；多域跨桥能力作为后续扩展点预留，避免过度设计 |

## 1. 背景与目标

### 1.1 背景

现有 dbridge 动态库（Qt 5.12.12 + C++17 + QtSql + QXlsx + SQLite）已具备一套 Profile 驱动的 Excel↔SQLite 导入导出能力，包括：基于 Profile 的字段映射与类型声明、ON CONFLICT Upsert 写入、三级校验（结构 / 类型 / 业务约束）、row-lookup 预取、fkInject 外键注入、反向 lookup 以及时间类型的显式处理。这些能力解决了"表格与库之间的受控搬运"问题，但尚不具备多节点之间的数据同步能力。

本需求文档在该能力基线之上，将 dbridge 扩展为一套完整的 SQLite 同步工具：在保留既有导入导出门面的同时，新增跨节点的增量同步、差异比对与合并，以及围绕灾难恢复、防回声、确定性仲裁与可测试性的工程化保障。

### 1.2 总体目标

- **星型增量同步**：在单域星型拓扑（中心 A + 子节点 B/C/D）下，以 SQLite 变更集为载体实现节点间的增量数据同步，支持基线 + 增量双模式（详见 FR-3）。
- **可视化差异比对与合并**：提供差异对比 / 合并 API 与内存缓冲（暂存）合并能力，由宿主应用构建可视化界面完成人工裁决（详见 FR-9、FR-10）。
- **同步触发模型**：上行（子节点 B→中心 A）由人工发起，且由人工指定参与同步的数据记录集，工具自动补全其外键依赖闭包（已与中心一致的依赖父行自动剪枝、不重复传输）、按拓扑序（父先子后）并以 UPSERT 方式应用；下行（中心 A→域内其它子节点）在中心固化裁决后自动发起广播。详见 FR-16 / FR-17。
- **低带宽适配**：面向 2Mbps 量级的低带宽链路设计，变更集紧凑传输，支持断点续传与离线投递。
- **传输层无关**：同步逻辑不绑定具体传输实现，仅与 outbox/inbox 文件制品交互，传输通道可替换为任意第三方黑盒工具。
- **保留并扩展门面**：在不破坏现有 dbridge 导入导出能力的前提下，新增一套精简的导入导出门面（详见 FR-13），既有 Profile 驱动能力保持向后兼容。

### 1.3 非目标（YAGNI）

下列内容在本版本中明确不做，以控制范围、避免过度设计：

| 非目标项 | 说明 |
| --- | --- |
| DDL / Schema 变更传播 | 表结构变更不随同步通道自动传播，由运维以受控方式统一发布；同步侧仅做 schema 版本校验与隔离，详见 FR-7。 |
| 多域跨桥（A↔H） | 多中心 / 跨域桥接不在本版本实现，仅在域拓扑上预留扩展点。 |
| CRDT | 不引入无冲突复制数据类型；冲突处理采用确定性仲裁 + 人工裁决路径。 |
| 自建网络协议 | 不设计专有网络传输协议；传输层为黑盒第三方工具。 |
| 分布式共识 | 不实现 Paxos/Raft 等分布式共识算法；中心节点为权威源。 |
| 完整 GUI 程序 | 工具仅提供 API/库能力，不交付完整图形界面应用，场景 2 的界面由宿主自建。 |
| 物理页 / 文件级复制 | 不做 SQLite 数据库文件的物理页或整文件级复制；以逻辑变更集为同步粒度。 |
| 非 SQLite 数据库同步 | 仅面向 SQLite；不支持其它关系型 / 非关系型数据库的同步。 |

## 2. 术语表

| 术语 | 定义 |
| --- | --- |
| 中心节点(Center) | 单域星型拓扑的中心,聚合各边缘子节点上行的变更并向下广播收敛后的载荷;通常为 dbridge 集成 SQLite 同步的权威汇聚点(场景中的 A 端)。 |
| 边缘子节点(Edge) | 围绕中心的叶子节点(场景中的 B 端),本地产生变更并由人工选择性上行推送,同时自动接收中心下行的载荷。 |
| 载荷/制品(Payload/Artifact) | 一次同步实际传输的数据单元,封装一批变更集及其溯源元数据、schema 指纹/版本与父引用,作为节点间交换的物理制品。 |
| 变更集(Changeset) | 对一行或一批记录的逻辑变更集合(插入/更新/删除),是 sync_changelog 与增量同步的基本处理与传播单位。 |
| 逻辑序号(seq) | 单调递增的逻辑序列号,标识变更在 sync_changelog 中的相对顺序;增量同步以「seq > 锚点」拼接(concat)选取待传变更。 |
| sync_changelog | 记录所有本地变更的变更日志表,字段含 seq、origin、schema 指纹、版本、父引用等,是增量同步与溯源裁决的数据来源。 |
| 基线同步(Baseline) | 冷启动时一次性全量同步,将对端数据完整对齐到统一起点,后续切换为增量模式。 |
| 增量同步(Incremental) | 在基线之上仅传输锚点之后(seq > 锚点)新增变更集的同步模式,避免重复全量。 |
| 同步锚点(Sync Anchor) | 标记某对端已成功接收/应用到的 seq 水位线;增量从锚点之后开始,锚点按 ACK 前移。 |
| 溯源元数据(Provenance) | 随变更集携带的来源信息(origin、schema 指纹/版本、父引用等),用于幂等、冲突仲裁与防回声判定。 |
| origin(来源/起源节点,不可变) | 变更最初产生的节点标识,在变更全生命周期内不可变,是防回声、规范序仲裁与幂等向量的分组键。 |
| origin 优先级(rank) | 为各 origin 预设的优先级排名,多源冲突时按 rank 高低构成规范序进行确定性仲裁。 |
| 变更集变基(Rebase) | rebaser 对冲突变更重新裁决并固化结果后再广播,使各节点在统一裁决下收敛到一致状态。 |
| Schema 指纹(Fingerprint) | 对表结构计算的摘要值,随载荷携带以快速识别 schema 是否一致,不匹配则触发漂移处理。 |
| Schema 版本 | 标识 schema 演进代次的版本号,与指纹配合用于漂移判定、隔离区版本标记与兼容性校验。 |
| ACK(确认) | 对端对已成功接收/应用变更的确认信号,据此将该对端的同步锚点前移。 |
| applied-vector(已应用向量) | 按 origin 维护的已应用 seq 水位向量,用于跨源幂等去重,丢弃已应用过的变更集。 |
| 隔离区(Quarantine) | 当检测到 schema 漂移(指纹/版本不匹配)时,临时隔离暂不可安全应用的变更并打版本号,待解决后再处理。 |
| 死对端逐出(Eviction) | 当某对端长时间无 ACK 或失联超过阈值时,将其从活动同步集合中逐出,避免无效保留与资源占用。 |
| 比对会话(Comparison Session) | 场景 2 中用于对照双方数据差异的短命会话,在其生命周期内计算待同步差异。 |
| 内存暂存缓冲(Staging Buffer) | 比对会话期间在内存中暂存的待处理变更/差异缓冲区,会话结束即释放,不落地为持久状态。 |
| Profile | 同步行为的可配置档案,声明显式类型、导出列序、反向 lookup、origin 优先级等参数,驱动不同环境下的同步策略。 |
| 同步触发模型(上行人工发起/下行自动发起) | 区分两个方向触发方式的模型:上行(边缘→中心)由人工显式发起选择性推送,下行(中心→边缘)由系统自动发起广播收敛。 |
| 上行选择性推送(人工指定记录集同步到中心) | 由人工挑选边缘端特定记录集,经 UPSERT 上行同步到中心的推送方式,而非全量自动上行。 |
| 外键依赖闭包(被选记录递归引用的父记录传递闭包,随选择集一并同步) | 对选择集中记录沿外键递归求解其引用的全部父记录所构成的传递闭包,随选择集一并上行以保证引用完整性。 |
| SyncSelection(上行选择集) | 上行推送时人工界定的待同步记录集合,可按「表 + 主键」枚举或「表 + 谓词」条件方式选择,并据此展开外键依赖闭包。 |
| 依赖一致性剪枝(Consistency Pruning) | 上行打包前对外键依赖闭包内父行做一致性判定,将「已与中心一致(行内容指纹相等)」的父行从载荷剪除、不重复传输,只传中心缺失或内容不一致的父行;是叠加在 UPSERT 幂等之上的纯带宽优化,存疑则保守纳入。仅作用于依赖闭包,不剪人工直选记录。 |
| 中心已知指纹(last-known-center fingerprint) | Edge 侧按 `(表, 主键)` 维护的"中心当前持有该行内容"的强哈希指纹,**只由中心权威制品(下行广播/基线)喂养、绝不由上行 ACK 盖章**;剪枝以"本地当前行指纹 == 中心已知指纹"为一致判据,二者均由 Edge 本地计算自比。 |
| 冻结清单(frozen manifest) | 上行选择性推送在短读快照内一次性算出的、持久化的拓扑有序 `(表, 主键)` + 指纹列表;算完即释放读快照(护 WAL),作为分片续传(push-id)的来源,语义为"闭包结构冻结、内容取打包时最新"。 |
| push-id / chunk-seq | 一发选择性推送的唯一标识(push-id)与其内分片的顺序号(chunk-seq);中心按 push-id 记分片应用进度、按 chunk-seq 顺序应用(父片不晚于子片),全片 ACK 才算该发"已应用"。 |
| streamEpoch / baselineGeneration | 载荷与 ACK 携带的代际号;outbox 坍缩成基线后递增,接收端据此丢弃/隔离早于当前代际的旧增量,杜绝基线前后混用。 |
| 静默(quiescence) | 域内无新变更、无在途载荷的稳定态;用于"防回声/最终一致"验收(静默后无新载荷、各节点校验和相等)。 |
| 前台操作面 / 后台常驻面 | 前台 = 宿主主动发起、受 `E_BUSY` 单活动门控的操作(syncSelected/手动 sync/import/export);后台 = 不受 E_BUSY 约束的常驻 worker(inbound 应用/ACK/下行广播/缓存盖章)。见 FR-11 §11.5。 |
| 单写者 worker | 引擎自有的串行写线程,所有写(inbound apply / 前台 capture / 广播打包)排同一队列入 SQLite 单写者,杜绝写写并发。 |
| ChangesetPayload / SelectionPushPayload | 两类载荷(F-02):前者 body 为压缩 SQLite changeset(自动增量/下行,经 `apply_v2`);后者 body 为冻结清单+行快照+recordKind+push-id+chunk-seq(上行选择性推送,经 UPSERT)。 |
| 完成哨兵(.ready) | 载荷主文件写完后再落地的标记文件;接收端先见哨兵、再读主文件,防读半截。 |

## 4. 功能需求

本章以功能需求(Functional Requirement,FR)形式定义 SQLite 同步工具的核心行为。各 FR 编号在全文档稳定唯一,跨节引用一律采用 FR-编号或章节号。本章按数据在系统中的流转链路组织,顺序如下:

1. **变更捕获**(FR-1):短命 session 捕获事务级变更并落地 sync_changelog。
2. **同步表范围与外部写检测**(FR-2):界定可同步范围,检测绕过 sync-aware 连接的外部写。
3. **载荷传输**(FR-3、FR-4):溯源元数据的封装与文件制品的写出、监听、ACK。
4. **应用与冲突**:增量 changeset 在对端的应用与冲突裁决(详见后续章节)。
5. **增量基线**:changelog/outbox 的截断与基线坍缩(联动 FR-8/FR-10)。
6. **广播**:一对多分发到多对端。
7. **死对端**:无响应对端的有界缓冲与降级(联动 FR-10)。
8. **状态机**:同步会话与对端的状态流转。
9. **场景 2**:典型业务场景的端到端串联。
10. **门面**:对上层应用暴露的统一同步门面 API。
11. **触发模型 / 上行选择性推送**:同步触发时机见 FR-16;上行数据范围不是全量增量,而是人工选择性推送 + 外键依赖闭包,见 FR-17——FR-2 的同步表集合仅界定"可同步范围",具体推送哪些记录由人工选择。

本章详述上述链路的前四个环节(FR-1 至 FR-4);其余 FR(应用冲突、增量基线、广播、死对端、状态机、场景 2、门面、FR-16 触发模型、FR-17 上行选择性推送)在后续章节展开。

### 4.1 变更捕获与变更历史

**FR-1**:工具必须以**短命 session(short-lived session)**为粒度捕获每一次本地事务/导入产生的数据变更,并在事务提交后立即将变更历史持久化到本地 `sync_changelog` 表。session 的生命周期严格绑定单个事务,不得跨事务复用。

**session 生命周期**:

| 阶段 | 时机 | 动作 |
| --- | --- | --- |
| 创建与附着 | `BEGIN` 之前 | 创建 SQLite session 对象,并对 FR-2 界定的同步表逐表 attach(空集合时 attach 全部用户表) |
| 变更累积 | 事务执行期间 | session 自动累积被 attach 表上的行级变更 |
| 收割(harvest) | **`COMMIT` 之前、同一事务内**(F-01) | 业务变更完成后、提交前调用 `sqlite3session_changeset()` 取出 changeset BLOB,并在**同一 SQLite 事务内**写入 `sync_changelog`,随后与业务变更**一并 `COMMIT`** |
| 销毁 | 收割完成 | 释放 session 对象,不跨事务复用 |

要点列表:

- **收割原子性**:changeset 取出与 `sync_changelog` 落地必须原子,二者要么同时成功要么同时回滚,杜绝"changeset 已生成但 changelog 未记录"的撕裂状态。
- **崩溃边界(F-01 整改)**:因 changeset 收割与 changelog 落地都在业务事务**提交点之前、同一事务内**完成,"业务已提交"与"changelog 已落地"是**同一个原子 `COMMIT`**——不存在"事务已提交但变更未捕获"的窗口。**例外声明**:若某实现确实无法做到同事务落地(只能在 `COMMIT` 后 harvest),则**必须显式承认存在"已提交未捕获"窗口**,并在重启恢复时通过 FR-2 的外部写检测(`data_version`/校验和)发现缺口、触发 re-baseline,不得声称零窗口。
- **sync_changelog 记录字段**:每条记录至少包含 `seq`(本地单调递增序列号)、`origin`(本节点标识)、`schema 指纹`、`schema 版本`、`父引用`(指向前驱变更,构成链式溯源)。
- **增量计算**:增量集合 = `seq > 锚点(anchor)` 的所有 changelog 记录按序 concat 而成;锚点由对端 ACK 推进(详见 FR-4 与后续增量基线章节)。
- **构建前提**:本能力依赖 SQLite 的 Session 扩展宏(`SQLITE_ENABLE_SESSION`/`SQLITE_ENABLE_PREUPDATE_HOOK`)。若编译期未启用,工具在初始化阶段必须报 **E_SYNC_SESSION_UNAVAILABLE** 并拒绝进入同步模式,不得静默降级。
- **截断策略**:`sync_changelog` 不可无界增长,必须配套截断/压实策略;截断时机与基线坍缩规则联动 **FR-8**(增量基线)与 **FR-10**(死对端有界缓冲),本节不重复定义。

### 4.2 同步表选择与外部写检测

**FR-2**:工具必须支持以**表级粒度**配置参与同步的"同步表集合"。该集合为空时,默认全部用户表参与同步。所有针对同步表的写入必须经由**sync-aware 连接**完成,以保证变更被 FR-1 的 session 捕获。

> **关键加注——"可同步范围"≠"实际推送范围"**:FR-2 配置的同步表集合仅界定**可同步范围**(哪些表的数据"有资格"被同步)。上行(上传)的实际数据范围**不是全量增量**,而是**人工记录级选择性推送 + 外键依赖闭包**,详见 **FR-17**。换言之,FR-2 回答"哪些表可以同步",FR-17 回答"这一次具体推送哪些记录"。

要点列表:

- **表级粒度**:同步配置以表为最小单位;空集合语义等价于"全表纳入"。同步表集合的变更会影响 session 的 attach 行为(见 FR-1 生命周期表)。
- **同步表 eligibility(G-04)**:SQLite Session 对可捕获表有**硬性前提**——**无显式 PRIMARY KEY 的表 attach 不报错但变更被静默丢弃,虚表/视图不支持**。故同步表必须为**普通表(rowid 或 `WITHOUT ROWID`)且具显式非空主键(或配置声明的 sync key)**;生成列不进入写列与指纹编码;UPSERT 冲突目标须为**完整唯一索引**(partial / 表达式唯一索引不可作冲突目标)。`initialize` 在 attach 前**逐表校验**,任一不合格报 **`E_SYNC_UNSUPPORTED_SCHEMA`** 并拒绝进入同步模式,**绝不静默跳过**——否则"配置成功却从不捕获"即为静默漏同步。
- **sync-aware 连接强制**:业务层对同步表的所有写路径都必须走 sync-aware 连接,否则变更将绕过 session 而不被捕获。
- **外部写检测**:工具必须能检测到未经 sync-aware 连接的"外部写"(如其他进程、外部工具直接改库)。检测手段:

| 手段 | 机制 |
| --- | --- |
| `data_version` 探测 | 比对 SQLite `PRAGMA data_version`,值跳变表明存在本连接之外的写入 |
| 校验和比对 | 对同步表内容计算校验和,与上次已知基线比对,不一致即判定漂移 |

- **未捕获外部写的响应**:一旦检测到未被 session 捕获的外部写,工具必须发出告警 **W_SYNC_UNTRACKED_CHANGE**,并触发**基线重算**(重新建立校验和/同步基线),以避免后续增量基于失真的基线而扩散错误。
- **与上行推送的关系**:外部写检测保障"可同步范围"内数据的可信基线;但具体哪些记录进入上行载荷,仍由 FR-17 的人工选择性推送 + 外键闭包决定。

### 4.3 载荷与溯源元数据

**FR-3**:工具产出的每一个同步载荷(payload)= **公共头(溯源元数据)+ 类型特定 body**。载荷分两类(F-02,消除"全是 changeset"与"上行是 UPSERT 行数据"的矛盾):

- **`ChangesetPayload`**(自动增量 / 下行广播):body = 压缩 SQLite changeset BLOB,经 `sqlite3changeset_apply_v2` 应用(FR-5)。
- **`SelectionPushPayload`**(上行人工选择性推送,FR-17):body = **冻结清单 + 行快照 + `recordKind`(直选/依赖)+ `push-id` + `chunk-seq`**,以 UPSERT 应用,**不是** SQLite changeset、不走 `sqlite3changeset_apply_v2`。

两类共享同一公共头;公共头任一字段缺失,接收端必须判定 **E_SYNC_PAYLOAD_CORRUPT** 并拒绝应用。

**载荷必含字段**:

| 字段 | 含义 | 缺失后果 |
| --- | --- | --- |
| body(类型特定) | `ChangesetPayload`=压缩 changeset BLOB;`SelectionPushPayload`=冻结清单+行快照+`recordKind`+`push-id`+`chunk-seq` | E_SYNC_PAYLOAD_CORRUPT |
| `origin` | 产生该载荷的源节点标识 | E_SYNC_PAYLOAD_CORRUPT |
| `seq` | 源节点的单调序列号,用于对端排序与锚点推进 | E_SYNC_PAYLOAD_CORRUPT |
| `schema 指纹` | 同步表结构的指纹,用于接收端结构兼容性校验 | E_SYNC_PAYLOAD_CORRUPT |
| `schema 版本` | 结构版本号,辅助兼容性判定与迁移 | E_SYNC_PAYLOAD_CORRUPT |
| `父引用` | 指向前驱变更,维持链式溯源与顺序完整性 | E_SYNC_PAYLOAD_CORRUPT |
| 目标 / 路由标签 | 标识载荷目标对端或路由分组(支撑广播与寻址外包) | E_SYNC_PAYLOAD_CORRUPT |

要点列表:

- **完整性即合法性**:上表字段构成载荷的最小合法集(公共头 6 项 + 类型特定 body);接收端在应用前必须逐项校验,任一缺失即整体判 **E_SYNC_PAYLOAD_CORRUPT**,不做部分应用。
- **压缩要求**:changeset 在封装前必须压缩,封装格式需满足在 **2 Mbps** 传输带宽下的体积/吞吐约束,确保低带宽链路下可用。
- **溯源链**:`origin` + `seq` + `父引用`三者共同构成跨节点可追溯的变更链,是增量排序、去重与冲突裁决的元数据基础。
- **与人工选择性推送的衔接**:载荷内压缩 changeset 的内容范围由 FR-17 的人工选择 + 外键闭包决定;FR-3 只规定"封装格式与必含元数据",不决定"装入哪些记录"。

### 4.4 文件制品传输与 ACK

**FR-4**:工具必须以**文件制品(file artifact)**为传输介质完成载荷的投递与 ACK 回执,通过 **outbox(发件目录)/ inbox(收件目录)** 配合 **QFileSystemWatcher** 监听与**完成哨兵文件**实现可靠的半双工文件通道。工具**不负责**重试、断点续传与寻址路由——这些职责外包给第三方传输层。

**传输机制**:

| 环节 | 机制 |
| --- | --- |
| 写出 | 载荷文件写入本节点 outbox 目录;写入完成后再放置**完成哨兵文件(`.ready`)**,标记该制品已完整可读 |
| 监听 | 接收端以 **QFileSystemWatcher** 监听自身 inbox 目录;仅当对应 `.ready` 哨兵出现后才读取载荷,避免读到半截文件 |
| 释放连接 | 载荷发送(写出 + 哨兵就位)完成后**立即释放数据库连接**,不持有长锁,杜绝传输耗时拖住本地库 |
| ACK 回执 | ACK **复用同一文件通道**(经对方 inbox / 本方 outbox 往返),无独立回执协议;ACK **必须在 `ackMaxDelayMs` 内以专用 ack 制品发出(F-14)**——捎带在下一出站载荷元数据仅作优化,**不得替代最迟发送保证**(否则对端长期无出站载荷会使 ACK 永不发、被误判滞后/死亡);ACK 携带 `streamEpoch`(F-13)用于推进发送端锚点(见 FR-1 增量计算) |

要点列表:

- **完成哨兵契约**:载荷主文件与 `.ready` 哨兵分两步落地——**先写主文件、后置哨兵**;接收端**先见哨兵、再读主文件**。该顺序是防半截文件读取的核心契约,接收端不得在哨兵缺失时读取主文件。
- **职责分层(F-06)**:**字节级**搬运重试、断点续传、寻址/路由由**第三方传输层**承担;**语义级**重投由 **dbridge** 承担——ACK 缺失超时后**重新出箱(re-outbox)**、`push-id`/`chunk-seq` 幂等补齐(FR-17)、`sync()` 安全续做(FR-11)。dbridge 不做字节级重传,第三方不懂语义级幂等,两层职责正交,不冲突。
- **InboxWatcher 可靠性(F-12)**:`QFileSystemWatcher` 可能漏事件、合并事件、在网络盘上失效,故**不得作为唯一消费机制**;InboxWatcher 必须在**启动时、watcher 触发时、周期 timer** 三种时机扫描 inbox,**以 `.ready` 哨兵 + 持久清单为准消费**,watcher 仅作唤醒优化。
- **不持长锁**:发送后立即释放连接,确保文件传输的耗时不反压到本地数据库事务。
- **outbox 有界与基线坍缩**:每个对端的 outbox 必须**有界**;当某对端积压超界(典型为死对端,见 FR-10),outbox 内待发载荷必须**坍缩成单个待发基线**(以最新全量基线取代历史增量序列),防止无界堆积。坍缩规则与触发阈值联动 **FR-10**(死对端有界缓冲)。
- **坍缩代际(F-13)**:载荷与 ACK 须携带 `streamEpoch`/`baselineGeneration`;基线坍缩后**递增 epoch**。坍缩前的旧增量可能已被第三方搬到对端 inbox(或 `.ready` 已就位),故接收端**必须丢弃或隔离早于当前 epoch 的旧增量**,杜绝基线前后混用旧增量导致错乱;ACK 亦按 epoch 匹配。
- **第三方目录契约**:outbox/inbox 的目录位置、文件命名规范、哨兵后缀(`.ready`)构成与第三方传输层的接口契约;契约需稳定且文档化,以便第三方传输层据此搬运文件。
- **ACK 与锚点**:对端成功应用载荷后回送 ACK,发送端据此推进锚点,使后续增量(FR-1:`seq > 锚点`)收敛;锚点推进与增量基线截断(FR-8)联动。
- **状态分层(F-05)**:"**锚点(anchor)**"专指**发送端**的 outbound ACK 水位 `outbound_ack(peer, origin, seq)`,仅供发送端裁剪/重发/截断;**接收端**的幂等状态另用 `applied_vector(origin, seq)`(FR-5)。二者是**不同节点上的不同状态表**,全文不混用——发送端凭 `outbound_ack` 决定发什么,接收端凭 `applied_vector` 去重。

### 4.5 变更应用与冲突回调

**FR-5 需求描述**:接收端在收到经校验、解压后的 **`ChangesetPayload`**(SQLite changeset)载荷后,必须通过现有 Upsert+事务通道(句柄穿透 `sqlite3*`)调用 `sqlite3changeset_apply_v2` 执行落库,并注册冲突回调函数对每一类冲突按确定性规则裁决。上行 `SelectionPushPayload` 走 UPSERT、不走本路径(见本节加注、FR-17)。应用过程不得绕过现有写入通道直接操作底层文件,确保与既有触发器、约束、changelog 捕获逻辑一致。**(F-16)** 本期**不含传输层加密**,故无"解密"步骤;载荷仅做校验+解压。如需端到端加密,另立 NFR、接口与错误码。

**应用前置校验**:在调用 `sqlite3changeset_apply_v2` 之前,必须先做 schema 指纹校验(见 4.7 / FR-7)。若接收端表结构指纹与载荷声明指纹不一致,直接拒绝应用并返回 `E_SYNC_SCHEMA_MISMATCH`,不进入冲突回调阶段。

**冲突常量 → 裁决映射表**:冲突回调依据 `sqlite3changeset_apply_v2` 传入的冲突类型常量返回对应处置码,映射如下:

| 冲突常量 | 语义 | 默认裁决动作 | changelog/结果标记 | 错误码 |
|---|---|---|---|---|
| `SQLITE_CHANGESET_DATA` | 写写冲突(目标行存在但值与载荷预期不符) | `REPLACE` | 记 `W_SYNC_CONFLICT_REPLACED` | 无(警告) |
| `SQLITE_CHANGESET_CONFLICT` | 主键/唯一键冲突 | `REPLACE` | 记 `W_SYNC_CONFLICT_REPLACED` | 无(警告) |
| `SQLITE_CHANGESET_NOTFOUND` | 目标行不存在(UPDATE/DELETE 找不到目标) | `OMIT` | 跳过该变更 | 无 |
| `SQLITE_CHANGESET_FOREIGN_KEY` | 外键约束被破坏 | `ABORT` | 整体事务回滚 | `E_SYNC_APPLY_FK` |
| `SQLITE_CHANGESET_CONSTRAINT` | 其他约束冲突(NOT NULL/CHECK 等) | `ABORT` | 整体事务回滚 | `E_SYNC_APPLY_CONSTRAINT` |

**冲突裁决策略可配**:`REPLACE` 类裁决受冲突策略开关约束,支持三档:

- `SourceWins`(默认):写写冲突与主键冲突一律 `REPLACE`,以载荷(源)值覆盖目标,记 `W_SYNC_CONFLICT_REPLACED`。
- `TargetWins`:对 `DATA` / `CONFLICT` 类冲突返回 `OMIT`,保留目标现值,记 `W_SYNC_CONFLICT_REPLACED` 的对偶标记(保留侧),不覆盖。
- `Manual`:对 `DATA` / `CONFLICT` 类冲突暂存为待裁决项并记冲突明细,不自动落库,交由上层人工处理。

`FOREIGN_KEY` / `CONSTRAINT` 的 `ABORT` 行为不受策略开关影响,任何档位下均回滚并报错。

**权威下行豁免 ConflictPolicy(F-04)**:中心 rebase 后的**权威下行广播载荷**必须以 **`AuthoritativeApply`(强制 `REPLACE`)** 应用,**不受 `ConflictPolicy` 影响**;`ConflictPolicy`(`TargetWins` / `Manual` 等)仅适用于**非权威入站 changeset** 与**场景 2 本地比对/人工合并**。否则若某 Edge 配置 `TargetWins`/`Manual` 拒绝中心裁决,FR-9 的"下游强制收敛"将不成立、最终一致性被破坏。

**原子事务边界(R2)**:单次接收应用必须将以下三项操作置于**同一原子事务**内,任一失败则整体回滚,无部分提交。**R2 的"原子"作用域为单个 apply 单元（单 changeset / 单分片，C13）**,而非一发多片选择性推送的整体——多片推送的整体语义是"可续 + 幂等 + FK 安全 + 最终全有"(见 FR-17),不要求所有片落在同一事务:

1. **apply**:通过 `sqlite3changeset_apply_v2` 落库业务数据(含冲突回调裁决结果);
2. **applied-vector 推进**:按 `(origin, seq)` 推进已应用向量,实现幂等去重——已应用过的 `(origin, seq)` 重复到达时整体跳过,不重复落库;**严格连续应用(G-05)**:changeset 流仅当 `seq == applied_seq+1` 才应用,`seq <= applied_seq` 为幂等 no-op,`seq > applied_seq+1`(乱序/缺口)入 pending 缓冲待补齐、持续缺口触发 `E_SYNC_GAP` 回退基线——杜绝乱序到达下"高 seq 先到推进水位、低 seq 后到被误判已见"的丢更;**选择性推送分片**幂等另以 `(origin, stream_epoch, push-id, chunk-seq)` 计(经 `push_chunk_progress`),不与 changeset 单高水位混用;
3. **转发副本落 changelog**:将本次变更以转发副本形式写入本端 changelog,供下游继续传播。

三者共享同一事务句柄,提交点唯一。若 apply 阶段触发 `ABORT`(FK/CONSTRAINT),则 applied-vector 不推进、changelog 不落,事务回滚至应用前状态。

**加注(上行选择性推送,C12)**:上行人工选择性推送(FR-17)的载荷在中心 A 端**不走 changeset apply 路径,而以 `UPSERT` 语义应用**,且**不叠加 FR-6 rank 仲裁**(FR-6 只管自动路径)。按记录类别分两种算子:**人工直选记录 → `ON CONFLICT DO UPDATE`(人工选择即权威)**;**依赖闭包父行 → `ON CONFLICT DO NOTHING`(只补缺失、绝不覆盖中心父值)**。两者均不看 rank、不记 per-row owner。该路径(单片)同样纳入本节 R2 的单 apply 单元原子事务边界(apply + applied-vector + 转发 changelog)。

### 4.6 多源并发冲突的确定性仲裁

**FR-6 需求描述**:在分区恢复(partition recovery)等多源并发场景下,中心节点必须对来自不同 origin 的冲突变更做**确定性仲裁**:无论变更以何种网络到达顺序抵达中心,最终收敛态必须唯一且可复现。仲裁不依赖到达时序,而依赖预先配置的 origin 优先级与逻辑序号构成的规范序。

**典型冲突场景**:分区期间源节点 B 与源节点 D 各自离线修改了同一行 `row1001`:

| 时序 | 事件 |
|---|---|
| T0 | 网络分区,B、D 与中心 A 失联 |
| T1 | B 本地将 `row1001.value` 改为 `B_VAL` |
| T1' | D 本地将 `row1001.value` 改为 `D_VAL` |
| T2 | 分区恢复,B、D 的变更以**任意顺序**抵达中心 A |

朴素的"后到覆盖先到"(last-writer-wins by arrival)会导致终态依赖网络抖动,不可复现。FR-6 要求终态与到达顺序解耦。

**origin 优先级 rank**:拓扑配置为每个 origin 指定一个全局唯一的优先级 `rank`(数值越高优先级越高,或按配置约定),由中心节点作为**唯一仲裁者**持有。

**规范序应用**:中心 A 对冲突变更集按以下规范序(canonical order)排序后应用,使**最高优先级 origin 的值成为最终值**:

```
规范序排序键 = (origin 优先级 rank, 逻辑序号 seq)
```

- 先按 origin 优先级 rank 比较;rank 高者胜出,其对 `row1001` 的写入为最终值。
- 同一 origin 内部按逻辑序号 seq 决定先后。
- 排序与变更的物理到达顺序**完全无关**。

**"源节点优先"语义**:本节所称"源节点优先"= **高优先级 origin 胜**。若拓扑配置 `rank(B) > rank(D)`,则无论 B、D 变更孰先抵达,`row1001` 终态恒为 `B_VAL`。

**配合 FR-9 rebase 广播收敛**:中心 A 完成规范序仲裁、确定 `row1001` 终值后,按 FR-9 将仲裁结果以 rebase 形式重新广播给所有源节点(含未胜出的 D),使全网在 `row1001` 上收敛到同一终态;落败侧 D 的本地值被中心权威值 rebase 覆盖。

**可测断言**:仲裁的确定性必须可测——

> 给定同一冲突变更集 `{B:row1001=B_VAL, D:row1001=D_VAL}`,以两种相反的到达顺序(B 先 / D 先)分别喂入中心 A,断言两次运行后 `row1001` 的终态**完全相同**(均为高优先级 origin 之值),且全网 rebase 广播后各源节点收敛至该同一终态。
>
> **跨批/晚到强化(G-01)**:可测断言必须覆盖**高 rank 先到并提交、低 rank 后到且分属不同 apply 批次**的场景——断言低 rank 后到**不得覆盖**已提交的高 rank 值("到达序无关"不止于"同批同时仲裁",更要在隔时分批到达时成立)。实现侧据此须**持久化逐行胜者状态**(每行当前胜者的 `origin/rank/seq`),冲突回调凭此裁决 REPLACE/OMIT,而非依赖 `apply_v2` 默认 REPLACE(默认 REPLACE 看不到当前行胜者来源,会让低 rank 后到者覆盖高 rank,违反本 FR)。

### 4.7 Schema 版本校验与隔离

**FR-7 需求描述**:同步通道**不承载 DDL / schema 变更的传播**(DDL/schema 传播为非目标,见 1.3);但每条同步载荷必须携带一个**单调递增的 schema 版本号**,接收端依据本端 schema 版本与载荷 schema 版本的关系决定接受、隔离或反向触发迁移,确保结构不一致时不崩溃、不静默写坏数据。

**载荷 schema 版本**:每条 changeset/patchset(及 FR-17 上行推送载荷)在元数据中携带发送端当时的单调 schema 版本号 `schema_ver`。该版本号随结构迁移单调递增,不随业务数据变更而变化。

**版本关系处置表**:接收端比较本端 `local_ver` 与载荷 `schema_ver`:

| 版本关系 | 含义 | 处置动作 | 错误码/后续 |
|---|---|---|---|
| `local_ver == schema_ver` | 结构一致 | 正常进入指纹校验与应用(见 4.5) | — |
| `local_ver < schema_ver` | 接收端结构落后于载荷 | **quarantine 隔离**该载荷,不应用 | 报 `E_SYNC_SCHEMA_MISMATCH`;接收端迁移到 `≥ schema_ver` 后**自动重放/re-baseline** 隔离区载荷 |
| `local_ver > schema_ver` | 发送端结构落后 | 由**发送端**先迁移并 re-baseline 后重发 | 通知发送端迁移;发送端迁移后 re-baseline(见 FR-8) |

**表存在性差异(交集同步)**:当收发双方声明的表集合不完全一致时:

- 仅同步**双方都声明且都实际存在**的表(表集合交集)。
- 一方缺失的表所对应变更进入 **quarantine 隔离**,不导致整体同步崩溃。
- 隔离的缺表变更在该表于接收端补齐后,按隔离区重放规则恢复应用。

**迁移后强制 re-baseline**:任一端因 schema 版本不一致触发迁移后,**必须强制 re-baseline**(重建基线快照并重新建立 applied-vector 基点,见 FR-8 与第 8 章),以避免迁移前后 changelog 序列与新结构错位。re-baseline 完成后,隔离区中版本已满足条件的载荷自动重放。

**指纹兜底**:schema 版本号校验通过后,仍需在应用前执行 4.5 所述的 **schema 指纹校验**作为兜底——即便 `schema_ver` 相等,若结构指纹不一致(例如版本号管理失误或带外改表),仍拒绝应用并报 `E_SYNC_SCHEMA_MISMATCH`,防止版本号与实际结构脱节导致写坏数据。

### 4.8 基线/增量双模式与锚点

**FR-8 需求描述**

同步引擎须在每次同步会话开始时,依据接收端当前持有的 per-peer 锚点(anchor)与源端 changelog 状态**自动判定**采用「增量模式」还是「基线模式(全量)」,无需运维手工指定模式。增量是常态高效路径;基线是兜底安全路径。判定一旦命中任一回退触发条件(见下表),即降级到基线,保证收敛正确性优先于带宽效率。

要点:

- **增量模式(默认)**:源端自接收端锚点对应 seq 之后的 changelog 区间打包为 changeset 增量载荷发送。仅当锚点有效且区间连续(无缺口)时成立。
- **基线模式(全量)**:源端导出当前数据库快照作为基线载荷;载荷头部携带源端**当前 max-seq**,接收端应用基线后即以该 max-seq 作为新锚点起点(见下「锚点处理」)。
- **回退基线触发条件**(任一成立即回退):

| 触发条件 | 检测点 | 关联错误/告警码 |
|---|---|---|
| 缺锚点 | 接收端无该 peer 的 anchor 记录(首次同步/锚点丢失) | —(正常首同步) |
| 锚点滞后越界 | 接收端锚点 seq < 源端 changelog **min-seq**(增量起点已被截断,出现缺口) | `E_SYNC_GAP` |
| 迁移后重对齐 | schema 迁移导致 changelog 不可跨版本复用,须重新对齐基准 | — |
| 运维强制 | 运维显式要求重建基线(数据校验/灾备恢复等) | — |

**基线载荷与新锚点**

- 基线载荷 = 数据库快照 + 头部 `source_max_seq`。
- 接收端应用基线成功后,将该 peer 的锚点直接置为 `source_max_seq`,后续增量从此点续传。这保证基线与增量交接处零缺口。

**锚点 per-peer 按 ACK 前移(指向 FR-9)**

- 锚点按对端(per-peer)独立维护,**不**全局共享。
- 锚点前移采用 ACK 确认机制:接收端应用 changeset/基线成功后回吐 ACK,发送端**收到 ACK 才前移**该 peer 锚点;未 ACK 的区间视为未确认,锚点保持原位。前移规则与广播路径下的锚点语义统一,详见 §4.9(FR-9)锚点按 ACK 前移条目。

**changelog 截断约束(联动 FR-10)**

- changelog 截断**绝不**早于「最落后对端仍需的最小 seq」:截断水位 = min(所有活跃 peer 锚点 seq)。
- 若某对端长期未 ACK 致其锚点远落后,触发截断压力时**不得**直接丢弃其所需区间;该对端被标记 `re-baseline`(下次同步强制走基线模式),由 FR-10 的 changelog 保留/截断与 re-baseline 标记逻辑统一处置。
- 截断与 re-baseline 标记的联动细则见 FR-10。

**大库阈值**

- 当基线快照体积超过阈值告警 `W_SYNC_BASELINE_LARGE` 时,基线传输不走单次内联通道,改由第三方传输通道续传(分片/断点续传),避免单会话超时与内存峰值;告警仅提示,不阻断同步。

**模式判定汇总表**

| 模式 | 触发条件 | 载荷 | 锚点处理 |
|---|---|---|---|
| 增量(默认) | 锚点存在 且 锚点 seq ≥ 源 changelog min-seq 且 区间连续 | 锚点之后区间的 changeset | 收 ACK 后前移至已确认区间末 seq |
| 基线(全量) | 缺锚点 / 锚点 seq < min-seq(`E_SYNC_GAP`) / 迁移后重对齐 / 运维强制 | DB 快照 + 头部 `source_max_seq`(大库经第三方续传,告警 `W_SYNC_BASELINE_LARGE`) | 应用成功并 ACK 后,锚点置为 `source_max_seq` |

### 4.9 星型广播、变基与防回声

**FR-9 需求描述**

在星型拓扑(中心 + 子节点 A/B/C/D…)下,中心节点收到某子节点(如 B)的上行变更并合并后,须向**域内其它子节点**(如 C、D)派生**下行广播载荷**,使全域收敛到同一裁决结果。下行广播采用「保留 origin + rebaser 固化裁决 + 路由防回声」三件套,保证不铸新来源、不产生回声风暴、下游强制收敛到中心裁决。对等中心间横向同步本期**不实现**,仅在协议层预留扩展位。

> 加注:本 FR 描述的中心→域内其它子节点的下行广播,由**中心自动发起**——中心收到并合并上行变更后自动派生下行载荷;**上行变更**(子节点→中心)由人工触发,见 FR-16,不属于本自动广播范围(对应共识 N2)。

要点:

- **转发保留 origin**:中心转发 B 的变更时,changeset 内保留其原始 origin(=B),**不**铸造新的 origin。应用远端变更时不生成本地新 origin、挂起本地捕获,origin 标记仅追加进 changelog 供后续转发。
- **rebaser 固化裁决**:中心以 `sqlite3changeset_apply_v2` 的 rebase 输出作为权威裁决,将广播载荷变基到中心已合并状态;下游强制应用该「变基后 changeset」,从而收敛到与中心一致的裁决结果(冲突由中心按 origin 优先级规范序仲裁后再 rebase,见 FR-6)。

**路由防回声规则表**

| 规则 | 含义 | 目的 |
|---|---|---|
| origin 过滤 | 仅向 origin ≠ 对端的对端发送 | 不把变更发回其来源 |
| seq 过滤 | 仅发送 seq > 该对端锚点的区间 | 避免重发已确认变更 |
| 禁回推来源 | 严禁将变更回推其 origin 节点 | 杜绝来源回声 |
| 禁反推中心 | 子节点不向中心反推下行收到的变更 | 杜绝中心↔子节点回环 |
| 保留 origin | 转发不改写 origin | 防回声判定有稳定依据 |

**锚点按 ACK 前移**

- 广播路径锚点同样 per-peer 维护、按 ACK 前移:中心向 C/D 发送后,待 C/D 应用并回吐 ACK,中心才前移对应 peer 锚点;未 ACK 不前移、亦不触发对该 peer 的 changelog 截断(截断水位与 re-baseline 联动见 §4.8 / FR-10)。

**派生路径 ASCII 图(B 上行 → 中心 A 合并裁决 → 下行广播 C、D)**

```
            (上行,人工触发 FR-16)
        B  ───────────────►  A(中心)
                                │  合并 + rebaser 固化裁决
                                │  (保留 origin=B)
              下行广播(自动,FR-9 / N2)
                 ┌──────────────┴──────────────┐
                 ▼                              ▼
                 C                              D
   路由:origin≠对端 且 seq>对端锚点;禁回推 B;C/D 不反推 A
```

- 图注:A 收到 B 的上行(人工触发,FR-16)后,自动合并并以 rebaser 固化裁决,变基后向 C、D 派生下行广播(本 FR);广播保留 origin=B,路由按防回声规则表过滤;C、D 应用后回吐 ACK,A 据此前移 C、D 的 per-peer 锚点。

### 4.10 死对端逐出与存储边界

**FR-10:** 中心必须为每个对端独立追踪同步进度落后程度,以"落后阈值三档可配"区分健康/告警/逐出,并保证任意单一对端无论如何落后或失联都不得无界占用中心存储,即"单个死节点不得撑爆中心磁盘"。

#### 10.1 per-peer 落后度量与三档阈值

落后程度对每个对端独立计量,可从三个维度任配其一或组合,每个维度各设软阈值与硬上限两档,叠加"健康"档共三档:

| 度量维度 | 含义 | 软阈值越界 | 硬上限越界 |
| --- | --- | --- | --- |
| commit 数 | 该对端 `lastAckedSeq` 落后中心最新已广播 seq 的提交条数 | 进入告警 | 触发逐出 |
| 字节数 | 为该对端在 outbox 中保留的待发增量累计字节 | 进入告警 | 触发逐出 |
| 时长 | 距该对端最近一次新鲜 ACK 的墙钟时长(存活性判定见 10.3) | 进入告警 | 触发逐出 |

阈值具体数值由中心侧配置项给定,三个维度均可缺省关闭;任一启用维度越界即按"取最严"原则升档。

#### 10.2 阈值档位与动作

| 阈值档位 | 触发条件 | 中心动作 |
| --- | --- | --- |
| 健康 | 所有启用维度均在软阈值内 | 正常为该对端保留增量、正常推进截断水位 |
| 软阈值(告警) | 任一启用维度越过软阈值、未达硬上限 | 发 `W_SYNC_PEER_LAGGING`(告警运维,见 10.4);仍继续为其保留增量 |
| 硬上限(逐出) | 任一启用维度越过硬上限 | 发 `E_SYNC_PEER_DEAD`;执行逐出动作集(见 10.3) |

#### 10.3 逐出动作集与存活性判定

对端被判定为 `E_SYNC_PEER_DEAD` 后,中心一次性执行下列动作:

1. **停止为它保留增量** —— 不再向其 outbox 追加待发数据,已保留增量可释放。
2. **对全域其余对端恢复正常截断** —— 该死对端不再参与"最慢对端"截断水位计算,中心日志/增量截断对全域其余对端恢复正常推进,从而回收被其长期占用的存储。
3. **标记回归时强制 re-baseline** —— 死对端被打上回归标记;若它日后重新出现,不得断点续传,必须强制走全量 re-baseline 再入域(恢复路径见 10.5)。

**存活性判定(无独立心跳):** 对端是否存活完全由其 ACK 新鲜度推断,中心不发送也不依赖任何独立心跳通道。最近一次新鲜 ACK 的墙钟时长即为 10.1 的"时长"维度度量来源。

#### 10.4 outbox 有界与溢出坍缩

每个对端的 outbox 必须容量有界(条数/字节上限由配置给定):

- outbox 未满时,按序追加该对端待发增量。
- outbox 即将溢出时,**坍缩成单个待发基线** —— 丢弃该对端积压的全部离散增量,代之以一个"待发基线"标记;该对端下次取数时直接走基线全量,而非逐条增量。
- 坍缩使任意单一对端的 outbox 占用被钳制在常数级上限,与积压增量条数解耦,这是"单个死节点不得撑爆中心磁盘"的底层保障;落后维度持续越界仍会按 10.2 升档直至逐出。

#### 10.5 死对端恢复路径

死对端不会自行恢复;恢复必须由运维介入,流转如下:

```
E_SYNC_PEER_DEAD(逐出,带回归标记)
        │  运维介入让其回来
        ▼
   重新连入中心
        │  命中回归标记 → 拒绝断点续传
        ▼
   强制 re-baseline(全量)
        │  基线对齐完成、清除回归标记
        ▼
   重新入域(恢复正常 Capturing→Exporting→等 ACK 循环)
```

所有档位迁移(健康↔告警、→逐出、逐出→回归→重新入域)均产生面向运维的告警/审计事件,供运维感知与决策。

### 4.11 同步状态机

**FR-11:** 一轮同步必须按确定的状态机流转,对外暴露稳定的阶段语义与可观测量;终态仅 `Completed`/`Stopped`/`Failed` 三者,任意非终态阶段均可被 `stop()` 协作式中止,而 `sync()` 因 ACK/锚点幂等可安全重跑。

#### 11.1 一轮同步状态流转

```
                         ┌─────────────────────────── stop() ───────────────────────────┐
                         │            (任意非终态阶段协作式中止)                          │
                         ▼                                                                │
  Idle ──> Capturing ──> Exporting ──> Importing ──> [仅中心] Broadcasting ──> Completed  │
                │            │             │                  │                           │
                └────────────┴─────────────┴──────────────────┴───> Stopped  <────────────┘
                             │
                             └──(不可恢复错误,任意阶段)──> Failed
```

| 阶段 | 含义 | percent |
| --- | --- | --- |
| Idle | 未开始/空闲 | — |
| Capturing | 采集本地变更集 | 0..100 |
| Exporting | 已交 outbox、等待 ACK;在途传输为黑盒(见 11.3) | **-1** |
| Importing | 应用收到的对端/中心增量 | 0..100 |
| Broadcasting | **仅中心**:向各对端分发增量(经各自 outbox 与 ACK) | 0..100 |
| Completed | 本轮成功终结 | 100 |
| Stopped | 被 `stop()` 协作式中止 | — |
| Failed | 遇不可恢复错误而终结 | — |

#### 11.2 流转与终态规则

- **正常路径:** `Idle → Capturing → Exporting → Importing →`(中心追加 `Broadcasting`)`→ Completed`;边缘节点无 `Broadcasting` 阶段。
- **`Exporting` 语义:** 即"已交 outbox 等 ACK",在途黑盒,故 `percent` 恒为 `-1`(不可估算进度),收到足额新鲜 ACK 方可离开本阶段。ACK 新鲜度同时驱动 4.10 的存活性判定与死对端逐出。
- **`stop()` 协作式中止:** 任意非终态阶段收到 `stop()` 均流向 `Stopped`,中止在阶段边界协作发生(非强杀),不破坏已提交锚点。
- **`sync()` 幂等重跑:** 因每轮以 ACK/锚点为幂等依据,被 `Stopped` 或瞬时失败后重跑 `sync()` 不产生重复应用,可安全续做。
- **不可恢复错误:** 任意阶段遇不可恢复错误直接流向 `Failed`。
- **终态唯一性:** 终态仅 `Completed`/`Stopped`/`Failed` 三者,进入终态后本轮不再迁移。

#### 11.3 dbridge 可观测量与在途黑盒声明

中心 dbridge 对每个对端仅暴露下列可观测量(其余传输细节不可见):

| 可观测量 | 含义 |
| --- | --- |
| `bytesPacked` | 已打包准备外发的字节数 |
| `bytesApplied` | 已成功应用入库的字节数 |
| `changesApplied` | 已成功应用的变更条数 |
| `conflicts` | 本轮检测到的冲突计数 |
| `lastAckedSeq` | 该对端最近确认的 seq(同时为 4.10 落后度量与存活性判定输入) |

**在途黑盒声明:** 传输在途部分(已离 outbox、对端尚未 ACK 之间)为黑盒,**不可观测**;故 `Exporting` 阶段 `percent=-1`,上层不得据此推断在途进度,只能以 `lastAckedSeq` 推进与 ACK 新鲜度作为唯一进度与存活信号。

#### 11.4 上行选择性推送同样经此状态机

边缘节点的**上行选择性推送(FR-17)** 不走独立路径,同样经 `Capturing → Exporting → 等 ACK`:采集待上行的选择性变更集进入 `Capturing`,交本地 outbox 后进入 `Exporting`(`percent=-1`,等中心 ACK),收到 ACK 后正常终结。其存活性、幂等与 `stop()` 协作式中止规则与本节完全一致。

#### 11.5 前台操作面 / 后台常驻面分离(C15)

上述"一轮同步"是宿主主动发起的**前台操作**;但 inbound 应用与下行广播是**连续后台职责**,不能套用同一把单活动锁。二者分面:

| 面 | 成员 | E_BUSY 约束 | 由谁驱动 |
| --- | --- | --- | --- |
| **前台操作面** | 宿主主动发起的 `syncSelected` / 手动 `sync` 一轮 / `startImport` / `startExport` | **受 C9 单活动门控**:重入即 `E_BUSY`、不排队不并发、导入导出互斥 | 宿主调用 |
| **后台常驻面** | inbound 收取+应用(`Importing`)、ACK 回吐、下行攒批广播(`Broadcasting`, C14)、一致性缓存盖章(C11) | **不受 E_BUSY 约束** | 引擎自有的单写者串行 worker |

规则:

- **写全部串行到 SQLite 单写者一条队列**(inbound apply / 前台 capture / 广播打包同排此队),天然无写写并发;`E_BUSY` **只拒"前台重复 start"**,绝不拒后台管线步骤。
- **长推送不挡后台**:C13 的"等 ACK"为**非排他等待**——它占着前台槽(故同一边缘此时不能再发第二个前台操作,合理),但**接收中心下行、刷新一致性缓存属后台**,照常推进,边缘不对中心更新失明。
- **中心角色**:中心对多边缘 inbound **后台串行排队应用、绝不以 `E_BUSY` 拒收**;下行广播是后台定时;中心的前台操作面基本闲置。一套引擎双面并存,中心/边缘只是"哪面用得多"的差异,接口不分叉。
- `state()` 暴露**前台操作态**;后台活动经 `progress()` 的 `bytesApplied` / `lastAckedSeq` / `changesApplied` 体现。

### 4.12 场景2 表级差异概览

**FR-12 表级差异概览**

需求描述:进入比对会话后,系统针对"同步表集合"(参与同步的表清单)逐表给出本地 B(左屏)相对中心 A(右屏)的差异状态,驱动 PNG 首层的表清单 + 右侧红/绿标签呈现。状态枚举如下表。

| 状态 | 标签色 | 含义 |
| --- | --- | --- |
| Identical | 绿 | 本地与中心该表内容一致(schema 指纹 + 行数 + 内容校验和全等;高水位不参与判等,G-06) |
| Different | 红 | 两侧均存在该表但内容有差异 |
| OnlyLocal | 红 | 仅本地存在(中心无对应或从未同步) |
| OnlyRemote | 红 | 仅中心存在(本地缺该表) |

要点:

| 维度 | 约定 |
| --- | --- |
| 零全量拉取 | 表级判定**绝不拉取任一侧整表数据**;仅依赖 schema 指纹 + 行数 + 内容校验和等轻量元数据(高水位仅信息量) |
| 中心侧来源 | 中心仅回传极小的"元数据字典"(每表:schema 指纹、行数、内容校验和;高水位附带但不判等),本地在内存中逐表比对得出状态 |
| 校验和来源(F-17) | 每表 `schema 指纹/行数/内容校验和` 为**中心同步状态表中随 changeset 增量维护的元数据**,**不允许**在比对请求时扫描业务全表;中心侧应答的 SELECT 行数同样有界(可经 onPrefetch 式钩子断言) |
| 判定逻辑(G-06) | **schema 指纹 + 行数 + 内容校验和三者全等 → Identical(绿)**;任一不等 → Different(红);单侧缺失 → OnlyLocal/OnlyRemote(红)。**高水位(已应用最大 origin_seq)不参与判等**——不同 origin 的 seq 空间相互独立、本地写无远端 origin_seq,两节点内容相同时高水位仍可不同,用其判等会假红/假绿;高水位仅作本地信息量/审计 |
| 快照一致性 | 概览所读本地侧数据须取自 FR-14 钉定的一致性读快照,避免与后续行级视图错位 |
| 结果非落盘 | 概览结果为内存计算产物,不写持久库;会话结束即可丢弃 |
| 双击下钻 | 任一红状态表可双击进入 FR-13 行级双屏差异视图;Identical 表亦允许进入(行级全部呈 Same) |

边界:OnlyLocal/OnlyRemote 行级下钻时,缺失侧整体以幽灵空行对齐(见 FR-13);元数据字典缺失或版本不识别时该表标记为需重新拉取而非误判绿色。

### 4.13 场景2 行级差异

**FR-13 行级差异**

需求描述:对某张表双击下钻后,系统给出行级差异明细,供宿主用左右双屏 QTreeView 并排呈现(左=本地 B,右=中心 A)。每行带行级状态;展开后逐列给出 local 值、remote 值与 changed 标志。行级状态枚举如下表。

| 行状态 | 含义 | 左屏(本地 B) | 右屏(中心 A) |
| --- | --- | --- | --- |
| Same | 两侧该行全列一致 | 实体行 | 实体行 |
| Added | 仅中心新增(本地无) | 幽灵空行 | 实体行 |
| Deleted | 仅本地存在(中心已删) | 实体行 | 幽灵空行 |
| Modified | 两侧均有但至少一列不同 | 实体行 | 实体行 |

列级明细:对 Modified 行逐列输出 `{列名, localValue, remoteValue, changed}`;`changed=true` 标识该列值不一致,供宿主高亮。Same 行各列 `changed=false`;Added 行 local 侧列值为空,Deleted 行 remote 侧列值为空。

要点:

| 维度 | 约定 |
| --- | --- |
| 远端侧物化范围 | 远端(中心 A)侧数据 = 已收到的载荷制品(changeset);**只物化 changeset 受影响行 + 其本地对应行**,绝不一次性拉取整表 |
| 整行浏览的有界取数(F-03) | 对 **OnlyRemote 整表**或 **Different 大表**需逐行浏览未变更行时,经**有界分页远端取行** `fetchRemoteRows(table, keysetPage, snapshotId)`(§5.4)按页(keyset 分页 + 快照号)拉取,**仍禁止一次性全表拉取**;不发起分页时仅显示 changeset 受影响行。场景 2 据此宣称"可完整下钻",而非"零远端读" |
| 未变更行读取 | 不在 changeset 内的行(即两侧均未变动)= 纯本地读,直接取自本地一致性读快照,不触达远端 |
| 幽灵空行对齐 | Added/Deleted 行在缺失侧以幽灵空行占位,保证左右双屏逐行锚定、行号对齐 |
| 主键对齐 | 行配对以主键(或同步键)为锚;Same/Modified 双侧主键相同,Added/Deleted 仅单侧出现 |
| 大表分页 | 行级视图支持分页(offset/limit),按页拉取/物化,避免一次性物化超大表;分页边界与一致性读快照绑定 |
| 快照一致性 | 行级所读本地数据取自 FR-14 钉定的一致性读快照,与 FR-12 概览同源,避免错位 |

边界:OnlyLocal 表下钻 → 所有行 Deleted(右屏整列幽灵空行);OnlyRemote 表下钻 → 所有行 Added(左屏整列幽灵空行);分页翻页期间快照保持不变(见 FR-14),翻页不重新取版本。

### 4.14 场景2 内存暂存与并发隔离

**FR-14 内存暂存与并发隔离**

需求描述:场景2 的全部比对/编辑数据仅驻留内存暂存缓冲,在用户显式 save() 前**绝不写入持久库**;比对会话须钉定一致性读快照并隔离并发的 inbox 自动应用,防止 TOCTOU(读到比对、决定到落盘之间被并发写改变前提)。

要点:

| 维度 | 约定 |
| --- | --- |
| 内存暂存缓冲 | 拉入的 changeset 受影响行、本地对应行、用户编辑结果全部进内存缓冲;持久库在 save() 前零写入 |
| 一致性读快照 | 会话开始钉定一致性读快照(WAL/`data_version`),FR-12 概览与 FR-13 行级、分页翻页全程读同一快照 |
| inbox 暂停排队 | 会话期间暂停"涉及被比对表"的 inbox 自动应用,改为排队;不影响未被比对的表 |
| 防 TOCTOU | 暂停排队确保比对→决策→落盘窗口内被比对表前提不被并发写改变 |
| 脚下变动检测 | 若钉定快照的 `data_version` 在会话期间发生移动(脚下被改),会话作废并报 `E_SYNC_STAGE_STALE`,要求重开比对会话 |
| 放行时机 | save() 或 discard() 完成后,放行此前排队的 inbox 自动应用 |

save() / discard() 语义:

| 操作 | 语义 |
| --- | --- |
| save() | 经现有 Upsert + 事务通道,以 `BEGIN IMMEDIATE` 将内存缓冲落盘,**作为普通 origin 本地写**;后续由场景1 增量机制自然传播。**明确取消旧"权威重塑/强优先推送"语义**——save 不再是特权覆盖,而是常规本地写入 |
| discard() | 直接销毁内存缓冲,**零落盘**、不改持久库;随后放行排队 inbox |

边界:save() 落盘若与放行后的 inbox 写产生冲突,按场景1 既有冲突/增量规则处理(本节不重复);`E_SYNC_STAGE_STALE` 作废后内存缓冲一并丢弃,用户须重开会话基于新快照重做比对。

### 4.15 精简批量导入导出门面

**FR-15**：在不改动现有阻塞式 `DataBridge::importExcel` / `DataBridge::exportExcel` API（保留向后兼容）的前提下，新增 `IBatchTransfer` 非阻塞门面，提供「启动后立即返回 + 调用方轮询」的批量导入导出能力。该门面**复用现有 ETL 引擎**（解析、类型转换、外键注入、错误归集等），不重写任何转换逻辑，仅在其外层包裹非阻塞调度与状态机（DRY）。

要点：

- **复用既有底座（DRY）**：`IBatchTransfer` 内部直接持有 `DataBridge&`（见 §5.5 工厂 `createBatchTransfer(DataBridge&)`），导入导出仍走现有引擎；其输入仍为 Types.h 中的 `ImportOptions` / `ExportOptions`，结果仍为 `ImportResult` / `ExportResult`，错误仍为 `RowError`，不引入平行类型。
- **非阻塞启动**：`startImport` / `startExport`（命名加 `start` 前缀，因 `import` / `export` 为 C++ 关键字/保留语义，详见 §5.5 注释）调用后立即返回 `bool`（是否成功受理），实际工作在后台推进；进度与结果经轮询获取。
- **单活动操作（生命周期）**：每个 `IBatchTransfer` 实例**同一时刻仅允许一个活动操作**。当已有导入或导出在 `Running` / `Stopping` 态时，再次 `startImport` / `startExport` 一律返回失败并置错误码 **`E_BUSY`**，**既不排队也不并发**。
- **导入导出互斥**：导入与导出共享同一「单活动」闸门——运行中的导入会阻止启动导出，反之亦然（与 §4.16 之外的同步门面 §C9 语义一致）。
- **`stop()` 协作中止**：请求当前活动操作进入 `Stopping` 并在 **chunk / 行边界**协作式中止（best-effort，无法打断单条 `execBatch`）；对 AllOrNothing 单事务导入，中止语义为**整体回滚**至操作前状态、不留半提交；停止后状态落在 `Stopped`。`stop()` 为 §5.5 的 C9 对称增补方法（与同步门面 `ISyncEngine::stop`（§5.3 ③）对称）。
- **`start*` 重置该方向轮询态**：每次成功 `startImport` 重置导入侧进度/错误/结果快照（`importProgress` / `importErrors` / `importResult`）；`startExport` 同理重置导出侧——保证轮询读到的是本次操作的数据而非上一次残留。
- **getter 加锁快照**：所有 getter（`importProgress` / `importErrors` / `importResult` / `exportProgress` / `exportErrors` / `exportResult` / `importState` / `exportState`）返回**加锁后的一致性快照**（值拷贝），调用方可安全在任意线程轮询。
- **纯轮询（回调延后）**：本期不提供进度/完成回调或信号，调用方按需轮询 `*Progress` / `*State` / `*Result`；回调式通知延后版本评估。
- **状态机**：`TransferState`：`Idle → Running → (Stopping →) Completed | Stopped | Failed`（见 §5.5 枚举）。进度 `TransferProgress` 中 `rowsTotal` 在不可预知总量时为 `-1`，`percent` 相应可能停留在保守值。

### 4.16 同步触发模型与上行选择性推送

**FR-16 同步触发模型**：明确 Center 与 Edge 之间两个方向的触发方式、数据范围与应用语义。

| 方向 | 触发方式 | 数据范围 | 应用语义 |
|------|----------|----------|----------|
| **上行** Edge → Center | **人工发起**（Edge 端显式调用，见 §5.6 `syncSelected`） | **人工指定记录集**（`SyncSelection`：表+主键集合 或 表+谓词 WHERE，多表可选）+ **自动递归外键依赖闭包**（`includeFkDependencies` 默认 `true`；**已与中心一致的依赖父行自动剪枝、不重复传输**，见 FR-17） | **UPSERT**（`ON CONFLICT DO UPDATE`）应用到 Center，拓扑序父先子后 |
| **下行** Center → Edge | **自动发起（去抖攒批）**：`broadcastIntervalMs`（攒批窗）或 `broadcastThreshold`（累积变更条数/字节）**先到先发**；空闲不发空包（C14） | **变基后 changeset**（rebase 后的增量变更集，窗口内对同一对端多次变更 concat 成一发） | 应用到各 Edge，冲突走**冲突回调**（按 `ConflictPolicy`） |

要点：

- **上行是非全量增量**：上行**不做全量推送**，仅推送调用方在 `SyncSelection` 中显式选定的记录及其外键依赖闭包（见 FR-17）；这是「人工、选择性、增量」语义，区别于下行的自动广播。
- **下行自动仍遵 FR-9**：下行自动广播仍遵循 FR-9 的核心保障——防回声（origin/changelog 过滤，不把节点自己的变更回灌）、rebase（应用前对本地未确认变更变基）、ACK 锚点（按已确认序号推进，断点续传幂等）。
- **敲定原「同步触发」待确认项**：FR-16 即对前期遗留的「同步由谁触发、何时触发」待确认项的正式敲定——上行人工、下行自动，二者解耦、各自独立推进轮询态。
- **下行触发 = 去抖攒批合并（C14）**：下行"自动"被精确化为可配、可测的确定触发——`broadcastIntervalMs` 或 `broadcastThreshold` 先满足谁就发；窗口内多次变更对同一对端 **concat 成一发**（复用 FR-10 outbox 有界/坍缩），**绝不一变更一发**，杜绝小载荷打爆 2Mbps；上行落地变更对中心即"普通新变更"，**折进同一攒批窗口**广播给其它子节点（不回推来源、不为其单独立即扇出）；窗口到点但无新变更则**不产出广播**（不发空包，与 C2 quiescence 一致）。

**FR-17 上行选择性推送与外键依赖闭包**：定义 Edge 端「指定数据 + 外键闭包」的上行 UPSERT 推送语义。

- **选择方式（二选一或并用，多表可选）**：

  | 选择方式 | 含义 | Builder 入口（见 §5.6） |
  |----------|------|--------------------------|
  | **表 + 主键集合** | 指定某表内若干主键对应的行 | `addRecord(table, pk)` / `addRecords(table, pks)` |
  | **表 + 谓词 WHERE** | 指定某表内满足 WHERE 表达式的行集合 | `addWhere(table, whereExpr)` |

- **递归外键依赖闭包（传递闭包）**：当 `includeFkDependencies` 为 `true`（默认）时，对每个被选行**递归求解其外键所指向的父行**，直至无新增父行（传递闭包）。该闭包计算**复用既有能力**：`SchemaIntrospector` 的 `foreign_key_list`（解析外键关系）、`TopoSorter`（拓扑排序）、`fkInject`（外键值注入/重映射），不另起炉灶。
- **依赖一致性剪枝（只传不一致的依赖父行）**：外键依赖闭包**仅为满足被选记录的引用完整性而附带**，故打包前对闭包内每个父行做一致性判定——**已与中心一致（行内容指纹相等）的父行直接从上行载荷剪除、不打包传输**；只有「中心缺失」或「中心存在但内容不一致」的父行才进入载荷并 UPSERT。判定与边界：
  - **判定依据 —— 缓存只认中心权威态（C10）**：以 Edge 侧维护的「中心已知指纹」（last-known-center fingerprint，按 `(表, 主键)` 记录）比对父行当前内容指纹，相等才判为一致。该缓存**只能由中心权威制品喂养**——即 Edge **已应用的下行广播 changeset 与 FR-8 基线全量**（二者均携带中心经 FR-6 仲裁后的权威态）；**严禁以"既往上行推送的 ACK"盖章**——在 FR-6 仲裁下 ACK 仅表示"中心已处理"，**不等于**"中心持有我的值"（B 推的 `P_B` 可能被高 rank 的 D 仲裁成 `P_D`）。判据为**正向证据制**：无中心权威指纹背书的父行一律按"不一致"纳入。"依赖值要成为权威"另走**显式 `addRecord` 选择**表达，与剪枝解耦。
  - **指纹原语钉死（C11）——本地自比、取消跨节点确定性**：剪枝比较的两个指纹**都由 Edge 用同一函数计算**——「当前本地父行指纹」vs「上次应用中心权威制品落到本地时即时盖章的指纹」，故只要求函数**在本地确定、随时间稳定**，**不要求跨节点逐位一致**（消除整类 affinity/编码/浮点跨平台坑）。指纹函数 = 对父行按 **schema 列序**取 `(存储类标签, 值字节)` 规范编码后的**强哈希（≥128 位，如 SHA-256 截断）**，显式区分 `NULL`/整型/浮点（IEEE-754 8 字节）/文本（UTF-8）/BLOB（原字节），杜绝跨类碰撞与误剪。**冷父行**（从未被中心权威制品覆盖、无缓存指纹）**永远纳入**（承接保守兜底）。**本地迁移某表即作废该表全部缓存指纹**（旧列布局算的指纹不可信，C17），由迁移后基线/广播重新盖章。
  - **保守兜底（正确性优先于带宽）**：无指纹记录、指纹无法确证或疑似过期时，**一律按「不一致」纳入载荷**。剪枝是叠加在幂等之上的**纯带宽优化**——任一父行无论剪与不剪，中心终态都由「纳入路径的应用算子（依赖父 `DO NOTHING` / 直选记录 `DO UPDATE`，见下条 C12）」唯一决定，**剪枝误判绝不会造成数据丢失或 FK 破裂**。
  - **剪枝范围边界**：剪枝**只作用于自动派生的外键依赖闭包**，**绝不剪除人工显式选定的记录**（`SyncSelection` 直选行是人工推送意图，无论是否一致都必传）。
  - **可观测**：本次推送的「闭包父行打包数 / 剪枝数」可经 `onPrefetch` 式计数钩子观测，支撑 §9「一致依赖不传输」断言。
- **拓扑序父先子后**：闭包内所有记录按外键依赖关系**拓扑排序**后应用——**父行先于子行**写入 Center，确保子行的外键约束在写入时已有对应父行。**(F-08) 本期仅支持可拓扑排序的无环 FK 闭包**；检测到 **FK 环 / 自引用** 无法拓扑定序时报 `E_SYNC_FK_CYCLE_UNSUPPORTED`（要支持环须 deferrable FK + 单事务内延迟校验，属后续扩展）。
- **UPSERT 应用 + 冲突按记录类别拆分（C12）**：上行落库**不叠加 FR-6 rank 仲裁**（FR-6 只管自动路径：changeset apply / 多源 / 下行广播）；上行按记录类别用两种算子：
  - **人工直选记录** → `ON CONFLICT DO UPDATE`：**人工选择即权威**，B 的选定值落中心（中心无则 INSERT、有则 UPDATE）。这正是 N1"人为指定数据同步到中心"的本意。
  - **依赖闭包父行** → `ON CONFLICT DO NOTHING`：**只为引用完整性补"缺失的父"，中心已存在即分毫不动**，绝不用被拖带的依赖值覆盖中心权威父值（与 C10 一致）。
  - 两种算子**均不看 rank、不记 per-row owner**（C7 在上行继续成立）。收敛性：B 直选推 `P_B` → 中心采纳并下行广播 `origin=B` 的 `P_B` → 全域收敛。
  - **边界**：两个边缘**并发人工推同一 PK** → 中心按到达应用顺序末者胜，并记 `W_SYNC_CONCURRENT_MANUAL_PUSH`（人工协调问题，仅留痕、不自动裁决）。
- **冻结清单取代长持读快照（C16）**：选择集与外键闭包在**一个短读快照内一次性算完**，物化成**持久化的冻结清单**（拓扑有序 `(表, 主键)` 列表 + 每行内容指纹），**随即释放读快照**——绝不为整发推送长持读事务（否则 WAL checkpoint 被钉住、WAL 无界膨胀）。语义为"**闭包结构快照冻结 + 内容取打包时最新**"：分片打包时读当前行内容并与清单指纹比对——**改了取最新值** + `W_SYNC_PUSH_ROW_DRIFTED` 留痕；**删了则剔除**（连带其必需父/子）+ 同告警。**(F-07)** 若剔除后任何**仍保留**的记录因父行缺失导致 FK 闭包不完整 → **整发失败 `E_SYNC_FK_CLOSURE_MISSING`、不得部分推送**（不容许"留下引用悬空的子行"）。冻结清单即 C13 续传的 push-id 持久源。
- **大闭包按拓扑序分片（C13）**：选择集+闭包超载荷预算时，按**拓扑序分片（父片绝不晚于子片）**，每片为幂等单元 `(push-id, chunk-seq)`、各自一个原子事务（依赖父 `DO NOTHING`、直选子 `DO UPDATE`）；中心按 `chunk-seq` 顺序应用，**父先落子后落、跨片不破 FK**。整发语义 = "**可续 + 幂等 + FK 安全 + 最终全有**"：载荷头带总片数，**全片 ACK 才算该选择集已应用**；中途中断靠 applied-vector 幂等**续传补齐**，终态与不中断一致。**硬上限**：单发选择集/闭包规模超可配上限 → `E_SYNC_SELECTION_TOO_LARGE`（逼人工缩小范围，一次手滑不得拖走半个库）。`push-id` 的"已应用片数/总片数"对外可观测。
- **撞 schema 迁移整发作废（C17）**：一发长推送横跨中心 schema 迁移时——迁移静默窗（FR-7 §8.2 step-1）须**先排空在途选择性推送**（等其完成或 `stop()` 取消）方可迁移；竞态兜底：若仍有押旧版本的片在迁移后到达，中心**按 push-id 整发拒收 `E_SYNC_PUSH_SCHEMA_MOVED`**（而非逐片隔离），标记该 push-id 失败、**不留半截脏态**；B 在新 schema 下**重新快照（新冻结清单）整发重推**，对同批 PK 的 UPSERT 幂等自愈，无需跨片回滚。
- **错误码**：
  - 选择集为空（无任何 `addRecord` / `addRecords` / `addWhere`，或谓词命中 0 行且无显式主键）→ **`E_SYNC_SELECTION_EMPTY`**。
  - 闭包内出现**悬挂父**（子行引用的父行在源库中缺失，无法补全外键闭包）→ **`E_SYNC_FK_CLOSURE_MISSING`**。
  - 单发选择集/闭包规模超硬上限 → **`E_SYNC_SELECTION_TOO_LARGE`**（C13）。
  - 推送跨中心迁移、押旧 schema 版本 → **`E_SYNC_PUSH_SCHEMA_MOVED`**（C17）。
  - 推送期间选中行被本地改/删 → 告警 **`W_SYNC_PUSH_ROW_DRIFTED`**（C16）；并发人工推同 PK → 告警 **`W_SYNC_CONCURRENT_MANUAL_PUSH`**（C12）。
- **复用同一传输底座**：上行选择性推送的**载荷打包、传输、ACK、锚点幂等**全部复用与下行同步相同的底座（FR-9 同步引擎的 outbox/inbox/ACK/序号锚点机制），不为上行单独实现一套传输栈。

## 5. 接口契约（草案）

### 5.1 同步公共类型

```cpp
namespace dbridge {
namespace sync {

// 节点角色
enum class NodeRole {
    Center,   // 中心节点：接收上行 UPSERT、自动广播下行 changeset
    Edge      // 子节点：人工发起上行选择性推送、接收下行广播
};

// 同步状态机（贯穿 §5.3 ISyncEngine::state ④ 与 SyncProgress::state）
enum class SyncState {
    Idle,         // 空闲，无活动同步
    Capturing,    // 捕获本地变更 / 计算选择集与外键闭包
    Exporting,    // 打包并发出载荷（传输在途不可观测，percent 恒 -1）
    Importing,    // 应用收到的载荷
    Broadcasting, // Center 向各 Edge 广播下行 changeset
    Completed,    // 本轮同步成功结束
    Stopped,      // 经 stop() 协作中止后回滚到的终态
    Failed        // 发生致命错误终止
};

// 冲突策略（与 FR-5 / FR-6 列级合并配合）
enum class ConflictPolicy {
    SourceWins,   // 默认：源端值覆盖目标端
    TargetWins,   // 保留目标端值
    Manual        // 交由调用方/冲突回调裁决
};

// 日志/错误严重级别
enum class Severity {
    Info,
    Warning,
    Error,
    Fatal
};

// 同步进度快照（经 §5.3 ISyncEngine::progress ⑤ 轮询）
struct SyncProgress {
    SyncState state = SyncState::Idle;
    // 传输在途不可观测：在 Exporting/等待 ACK 等阶段 percent 恒为 -1（不可估算）
    int percent = -1;
    qint64 bytesPacked = 0;    // 已打包字节
    qint64 bytesApplied = 0;   // 已应用字节
    int changesApplied = 0;    // 已应用变更条数
    int conflicts = 0;         // 累计冲突数
};

// 每对端（peer）的同步状态，随 SyncResult 返回
struct PeerSyncState {
    QString nodeId;
    qint64 lastAckedSeq = -1;  // 对端已确认的最大序号（ACK 锚点）
    qint64 lastSentSeq = -1;   // 已发往该对端的最大序号
    bool lagging = false;      // 是否滞后（超过软限）
    bool evicted = false;      // 是否已被驱逐（超过硬限）
};

// 同步日志条目（经 §5.3 ISyncEngine::logs ⑥ 轮询）
struct SyncLogEntry {
    qint64 epochMs;     // 纪元毫秒时间戳
    Severity severity;
    QString phase;      // 阶段标识（如 "capture"/"export"/"apply"）
    QString message;
};

// 同步错误（经 §5.3 ISyncEngine::errors ⑦ 轮询）
struct SyncError {
    QString code;       // 错误码，如 E_BUSY / E_SYNC_SELECTION_EMPTY / E_SYNC_FK_CLOSURE_MISSING
    Severity severity;
    QString phase;
    QString nodeId;     // 关联对端（如有）
    QString message;
};

// 同步结果汇总（经 §5.3 ISyncEngine::result ⑧ 获取）
struct SyncResult {
    bool ok = false;
    SyncState finalState = SyncState::Idle;
    int payloadsSent = 0;       // 发出的载荷数
    int payloadsApplied = 0;    // 应用的载荷数
    int changesApplied = 0;     // 应用的变更条数
    int conflicts = 0;          // 冲突数
    QList<PeerSyncState> peers; // per-peer 状态
    QList<SyncError> errors;    // 累计错误
};

} // namespace sync
} // namespace dbridge
```

## 6. 非功能需求（NFR）

本章针对同步引擎提出非功能约束。除特别说明外，每条 NFR 均沿用现有 dbridge 工程惯例，并与第 5 章功能需求（FR-）配套验收。

**NFR-1 架构最佳实践。** 同步引擎对外仅暴露纯抽象接口 + 工厂函数 + PImpl 实现隐藏，遵循依赖倒置（DIP）：上层依赖接口而非具体类。沿用既有的 `DBRIDGE_EXPORT` 导出宏、`dbridge` 命名空间与字符串错误码风格（见第 7 章），不引入异类约定。同步引擎与现有 ETL/导入引擎之间通过明确接口协作、互不侵入：同步引擎复用 ETL 侧的落库通道（见 NFR-3），但不反向依赖 ETL 的内部状态；ETL 引擎对同步引擎的存在无感知。门面层任一时刻仅允许一个活动操作，重入时返回 `E_BUSY`（见第 7 章）。

**NFR-2 KISS / YAGNI。** 同步制品保持最简：基于文件的 outbox/inbox 载荷 + 目录监听（轮询或文件系统通知），不引入消息队列、分布式共识、CRDT 等重型机制。多域路由、CDC 后端替换、传输适配等扩展点本期仅在接口层预留（见 NFR-9），不落地实现，避免过度设计。任何"将来可能用到"的能力一律不提前构建。

**NFR-3 DRY / SOLID。** 落库路径收敛到唯一通道：同步应用落库、ETL 导入落库、场景 2 的 save 落库、上行选择性推送落库共四处，统一复用同一 Upsert + 事务通道（复用现有 `SchemaIntrospector` / `TopoSorter` / `fkInject`），不得各自重写写入逻辑。上行外键依赖闭包构建复用 `TopoSorter` 与 `fkInject`（对应 FR 上行推送条目）。各组件遵循单一职责：变更捕获、打包、传输、应用、冲突裁决、路由各为独立职责类。遵循开闭原则，以下三处为可替换策略点：冲突裁决策略、传输适配、CDC 后端，均经由接口注入，新增策略不改动调用方。

**NFR-4 编码硬约束。** 单个函数体不超过 150 行；函数参数不超过 7 个，超出时以配置结构体或 Builder 承载入参。所有配置对象一律采用 Builder 模式构造，不暴露多参构造函数。代码须通过工程既定 `.clang-format` 与 pre-commit 钩子校验后方可提交。

**NFR-5 性能与低带宽。** 载荷以压缩二进制（Session changeset / patchset）承载；增量同步实现表级零拉取（无变更的表不产生载荷）；应用与捕获均采用分批提交，并配合手动 `wal_checkpoint` 抑制 WAL 膨胀。**低带宽达标的 dbridge 可测项是载荷字节体积（`bytesPacked` / `bytesApplied`），而非端到端传输时延**：传输信道为第三方黑盒、在途不可观测，dbridge 不对传输时延做承诺，仅对其产出的载荷体积负责。

**NFR-6 并发与线程模型。** 严格遵循 SQLite 与 Qt 的线程约束：`QSqlDatabase` / `QSqlQuery` 不跨线程使用；底层原生 `sqlite3*` 句柄严禁跨线程传递；写连接为单线程、SQLite 单写者语义；Session 录制与 changeset Apply 必须在持有该连接的线程内执行。对外的 getter（观测量、状态快照）须线程安全，返回加锁拷贝的瞬时快照，不暴露内部可变引用。

**NFR-7 可观测与可审计。** 输出结构化日志，每条至少含：时间戳、阶段（捕获/打包/传输/应用/裁决/路由）、来源节点与目标节点、本次变更条数、冲突处置结果。冲突按源优先 REPLACE 覆盖时，须以警告 `W_SYNC_CONFLICT_REPLACED` 留痕，保证审计可追溯。可观测量限定为 C6 约定的 `bytesPacked` / `bytesApplied` / `changesApplied` / `conflicts` / `lastAckedSeq`；传输在途为黑盒，不提供在途观测量。

**NFR-8 可靠性与隔离。** 变更集应用全程事务化：外键破裂（`E_SYNC_APPLY_FK`）或其它约束破裂（`E_SYNC_APPLY_CONSTRAINT`）一律回滚整个事务，绝不部分落盘。场景 2 在 save 之前于内存中暂存隔离，discard 时零回滚、不触碰数据库。inbox 消费以"完成哨兵"标记可消费，避免读取半写入的载荷。apply 落库、applied-vector 推进、changelog 写入须置于同一原子事务中，保证三者要么全部生效、要么全部回滚，不出现进度与数据不一致。

**NFR-9 可扩展性。** 为后续演进预留接口，本期不实现、仅保证接口不阻断后续接入：路由表预留多域（多业务域）路由能力，当前仅支持单域；CDC 后端预留可替换点，默认 Session 扩展、未来可切换触发器方案；传输适配预留可替换点，默认基于文件的 outbox/inbox、未来可切换 IPC 或网络传输。上述扩展点均以接口形式定义，本期实现填充默认策略即可，新增后端不得要求修改既有调用方（呼应 NFR-3 开闭原则）。

## 7. 错误码（扩展 dbridge::err）

同步引擎沿用现有 `dbridge::err` 字符串常量风格（错误码以稳定字符串常量定义于 `dbridge::err` 命名空间，而非枚举或整型码），便于跨进程载荷、日志与人工排障直接读取。本期在 `dbridge::err` 下新增"同步"区段，集中承载下列同步相关错误码与警告码；既有非同步错误码不变。严重性分级：Fatal（终止流程）、Error（隔离/回滚/告警但流程可控）、Warning（留痕提示，不中断）。

### 7.1 错误码（Error / Fatal）

| 错误码 | 含义 | 严重性 | 处置 |
| --- | --- | --- | --- |
| `E_SYNC_INIT` | 同步引擎初始化失败 | Fatal | 终止同步流程 |
| `E_SYNC_SESSION_UNAVAILABLE` | 当前 SQLite 未启用 Session 扩展 | Fatal | 终止流程，不可降级运行 |
| `E_SYNC_SCHEMA_MISMATCH` | schema 指纹 / 版本与本地不一致 | Error | 隔离该载荷，不应用 |
| `E_SYNC_PAYLOAD_CORRUPT` | 载荷损坏或元数据缺失 | Error | 丢弃并告警 |
| `E_SYNC_TRANSPORT` | outbox / inbox 的 IO 失败 | Error | 可重试 |
| `E_SYNC_APPLY_FK` | 应用变更集时外键破裂 | Error | 回滚整个事务（见 NFR-8） |
| `E_SYNC_APPLY_CONSTRAINT` | 应用变更集时其它约束破裂 | Error | 回滚整个事务（见 NFR-8） |
| `E_SYNC_NODE_UNKNOWN` | 路由目标节点未知 | Error | 拒绝路由该载荷 |
| `E_SYNC_STAGE_CONFLICT` | 场景 2 暂存合并冲突 | Error | 提示人工介入处理 |

| `E_SYNC_UNSUPPORTED_SCHEMA` | 同步表无显式主键 / 为虚表/视图/影子表 / 无可用 UPSERT 冲突目标 | Fatal | initialize 拒绝进入同步模式、不 attach（G-04） |

| `E_SYNC_BASELINE_FAILED` | 基线导出/应用失败 | Error | 基线事务回滚、保留旧锚点；提示重试/换路（G-09） |

## 8. 灾难恢复与运维规程

本章给出域内节点在异常态(冷启动、Schema 漂移、对端死亡、外部写入、隔离积压、进程崩溃)下的恢复路径与运维操作规程。所有恢复路径的终态收敛保证由第 6 章(锚点/截断)、第 7 章(确定性仲裁)共识承载;本章只描述触发条件、操作步骤与回退选择。各子节涉及的可观测断言见第 9 章。

### 8.1 冷启动与基线引导

适用对象:全新入域节点、从离线备份恢复的节点、以及任何被判定为「锚点缺失或落后于源 min-seq」(`E_SYNC_GAP`,见 C3)而无法以增量续传的节点。

| 触发场景 | 判定条件 | 引导动作 | 锚点初值 |
| --- | --- | --- | --- |
| 全新节点入域 | 本地无 `sync_changelog`、无 applied-vector | 向域请求基线全量快照 | 取快照携带的 `max-seq` 作为该 origin 锚点 |
| 从备份恢复 | 本地锚点 < 任一源当前 `min-seq` | 丢弃陈旧增量,转基线全量 | 基线快照 `max-seq` |
| 增量缺口 | 检出 `E_SYNC_GAP`(缺锚点 / 锚点 < 源 min-seq) | 回退基线模式(C3) | 基线快照 `max-seq` |

要点:

- 基线快照必须为「带 `max-seq` 标记的一致性切面」,引导节点以该 `max-seq` 初始化 per-origin applied-vector(见 C6),此后所有 `seq ≤ max-seq` 的载荷靠幂等去重视为 no-op(参见 9 章「幂等」条款)。
- 基线引导完成前,节点不得对外发布任何增量载荷,避免以不完整状态参与仲裁。
- 引导走基线模式而非增量,是 C3「缺锚点 / 锚点 < 源 min-seq → 回退基线」共识的运维落地。

### 8.2 Schema 协调迁移规程

DDL 变更不属于同步载荷目标(C4):同步通道只搬运数据载荷,schema 升级由全域受控规程驱动。每条载荷携带 schema 版本号;接收端版本 < 载荷版本时,载荷进入隔离区(quarantine)并报 `E_SYNC_SCHEMA_MISMATCH`(见 8.5、C4)。

运维步骤(全域协调,逐节点串行或灰度):

1. 冻结待迁移表的增量发布(进入运维静默窗),确保在途载荷收敛;**显式排空在途的多片选择性推送**(等其自然完成或对在途 push-id `stop()` 取消)——这是防"长推送横跨迁移→半截脏态"的第一道闸(C17)。
2. 在协调器登记新 schema 版本号(单调递增),作为迁移目标版本。
3. 逐节点执行 DDL 迁移,迁移后本地 schema 版本递增至目标版本。
4. 迁移完成的节点对该表强制 **re-baseline**(重算基线,见 C3),以新 schema 版本为载荷版本对外发布。
5. 隔离区中因旧版本积压的载荷,在本地版本 ≥ 载荷版本后自动重放(见 8.5)。
6. 全域所有节点版本对齐至目标版本后,解除静默窗。

要点:

- 版本号随载荷传播,接收端「版本 < 载荷 → 隔离」的判定使迁移期半新半旧的域不会错应用高版本载荷。
- 迁移后强制 re-baseline,避免以旧 schema 产生的增量与新 schema 混用导致漂移。
- **缓存按表作废(C17)**:节点本地迁移某表后,**立即作废该表全部依赖一致性缓存指纹**(旧列布局算的指纹不可信),由迁移后基线/下行广播重新盖章;重建前该表父行一律冷父必传,宁可多传不可误剪。
- **竞态兜底**:若押旧版本的推送片在迁移后仍到达,中心按 push-id 整发拒收 `E_SYNC_PUSH_SCHEMA_MOVED`、不留半截脏态;B 新 schema 重新快照整发重推,幂等自愈。

### 8.3 死对端逐出与恢复

对端存活性以 ACK 新鲜度判定(C8):ACK 持续未前移即视为滞后/死亡。逐出采用软硬双阈值。

| 阶段 | 触发条件 | 信号 | 系统动作 |
| --- | --- | --- | --- |
| 软滞后 | ACK 停滞超软阈值 | `W_SYNC_PEER_LAGGING`(告警) | 保留 outbox,继续重试;不截断该对端待 ACK 数据 |
| 硬死亡 | ACK 停滞超硬阈值 | `E_SYNC_PEER_DEAD`(错误) | 将该对端逐出存活集;其 outbox 有界坍缩为基线;锚点截断不再受其未 ACK 阻塞(C6/C8) |

恢复路径(运维让死对端回来):

1. 运维确认死对端恢复在线,显式发起「重新入域」。
2. 该节点按 8.1 走冷启动 / 基线引导(其本地锚点多半已落后于源 min-seq,触发 `E_SYNC_GAP` → 基线全量)。
3. 引导完成后重算基线(re-baseline),以基线 `max-seq` 重置 applied-vector。
4. 重新纳入存活集,恢复 ACK 新鲜度上报;域据其 ACK 重新约束锚点前移。

要点:逐出是为解除「单个死对端永久阻塞全域截断」的死锁;outbox 坍缩成基线,保证逐出后内存/磁盘有界。恢复必经 re-baseline,而非续传陈旧增量。

### 8.4 未捕获外部写入恢复

同步框架之外的进程(外部工具、手工 SQL)直接改库,绕过 changeset 采集,导致 `data_version` / 校验和与同步态不一致。

| 环节 | 机制 |
| --- | --- |
| 检测 | 比对 SQLite `data_version` 与上次同步记录的版本 / 表校验和,发现外部改动 |
| 信号 | `W_SYNC_UNTRACKED_CHANGE`(警告:存在未被同步捕获的外部写) |
| 恢复 | 触发基线重算(re-baseline):以当前库实际状态重建基线快照,刷新校验和与 `max-seq` 锚点 |

要点:外部写无法事后逆向拆解为增量 changeset,故唯一安全恢复是「以现状为准重算基线」,使本地态重新成为对外发布的权威切面;重算后 applied-vector 锚点随新 `max-seq` 前移。

### 8.5 隔离区(quarantine)重放

接收端 schema 版本 < 载荷版本时,载荷不丢弃也不应用,而是落入隔离区挂起(C4),并报 `E_SYNC_SCHEMA_MISMATCH`。隔离区是迁移期「半新半旧」域的缓冲带。

| 状态 | 条件 | 动作 |
| --- | --- | --- |
| 挂起 | 本地版本 < 载荷 schema 版本 | 载荷入隔离区,记录其 schema 版本;不应用、不 ACK |
| 自动重放 | 本地完成迁移,版本 ≥ 载荷 schema 版本 | 隔离载荷按 (origin, seq) 顺序重放应用,经 applied-vector 幂等去重 |
| 重放后对齐 | 重放可能与本地新基线冲突 | 按需对该表 re-baseline,确保终态与全域一致 |

要点:

- 隔离重放严格走 applied-vector 幂等通道(C6),重放期间重复 seq 视为 no-op,保证「至少一次 + 幂等 = 有效一次」。
- 若隔离载荷因跨多个 schema 版本而无法逐版本对齐,降级为按需 re-baseline(以基线全量替代逐条重放)。

### 8.6 进程崩溃恢复

崩溃恢复以 SQLite 自身 journal(WAL / rollback journal)为事务原子性基线,同步层不另设事务日志,只在其上叠加幂等语义。

| 数据形态 | 崩溃时机 | 恢复结果 |
| --- | --- | --- |
| 未 harvest 的 changeset | 事务尚未提交即崩溃 | 随未提交事务被 SQLite journal 一并回滚丢弃,不进入同步管道(无半截载荷) |
| 已提交未发布的增量 | 提交后、发布前崩溃 | 重启后从 changelog 重新采集发布,seq 连续不丢 |
| 已发布已 ACK 的数据 | ACK 落盘后崩溃 | 靠 per-origin applied-vector 幂等去重,重启重投视为 no-op,不丢不重复 |
| ACK 丢失 | 对端已应用但 ACK 未达 | 锚点不前移,源侧重发;接收端 applied-vector 幂等去重兜底,重复应用为 no-op |

要点:

- 「未 harvest changeset 随未提交事务丢弃」是刻意取舍——崩溃边界与 SQLite 事务边界对齐,杜绝半提交载荷污染同步态。
- 已 ACK 数据的「不丢不重复」由 applied-vector 承载,而非依赖 ACK 可靠送达;ACK 仅控制锚点前移(见 C6),其丢失只导致重发而非数据错误。

## 9. 可测试性验收准则

下表给出各核心同步性质的验收条款、可观测断言与阈值/数值。可观测断言优先复用既有测试钩子先例:`ImportService::onPrefetch` 计数钩子(注入计数 lambda 对底层 SELECT 计数),用于「零全量拉取」类断言。

| 条款 | 可观测断言 | 阈值/数值 |
| --- | --- | --- |
| 防回声(C2) | 域静默(quiescence)后无新增载荷发出;抓取任一节点 outbox,不存在 `origin = 对端` 的回推载荷(只发 `origin ≠ 对端 且 seq > 锚点`) | 静默后新载荷数 = 0;回推载荷数 = 0 |
| 最终一致 | 同步收敛后,各节点同步表内容校验和两两相等 | 校验和差异 = 0;收敛超时上限〔TBD:设计阶段定 / 责任人待指派〕 |
| 幂等(C6) | 同一 `(origin, seq)` 载荷重复投递,applied-vector 判定为 no-op,表行数 / 校验和不变 | 重投后状态变更次数 = 0 |
| 确定性仲裁(C7) | 同一组冲突载荷以两种不同到达顺序分别投递,收敛终态一致(origin 优先级规范序仲裁,与到达顺序无关) | 两序终态校验和相等(差异 = 0) |
| 零全量拉取 | 经 `onPrefetch` 式计数钩子断言:一次增量同步触发的底层 SELECT 行数有上界,且与同步表总行数无关(O(选择集 + 闭包),非 O(表) ) | SELECT 行数 ≤ 上界且与表行数无相关性;上界常数〔TBD:设计阶段定 / 责任人待指派〕 |
| ACK 锚点(C6) | 锚点前移当且仅当收到对应 ACK;构造「应用但未 ACK」场景,断言锚点不前移 | 无 ACK 时锚点位移 = 0;收到 ACK 后锚点前移量 = ACK 覆盖 seq 数 |
| 死对端逐出(C8) | 对端永不 ACK 且超硬阈值,断言:changelog 截断得以恢复(不再被该对端阻塞)+ 该对端被标记 re-baseline | 硬阈值〔TBD:设计阶段定 / 责任人待指派〕;逐出后截断恢复 = true |
| 场景 2 隔离(C5) | save 前对 `.db` 文件做物理写探针,断言无物理写;discard 后断言无残留(内存隔离态清空,磁盘无半截数据);越权访问陈旧 stage 报 `E_SYNC_STAGE_STALE` | save 前 `.db` 物理写字节数 = 0;discard 后残留记录数 = 0 |
| 上行外键依赖闭包完整性(N1) | 选择集 + 依赖闭包(UPSERT)对对端 A 应用后:无 FK 违例,闭包内父记录全部就位;空选择报 `E_SYNC_SELECTION_EMPTY`;存在悬挂父引用报 `E_SYNC_FK_CLOSURE_MISSING` | 应用后 FK 违例数 = 0;缺失闭包父记录数 = 0;空选择 → 抛 `E_SYNC_SELECTION_EMPTY`;悬挂父 → 抛 `E_SYNC_FK_CLOSURE_MISSING` |
| 上行依赖一致性剪枝(N1) | 构造「闭包父行在中心已一致」场景,经 `onPrefetch` 式计数钩子断言该父行**不进入上行载荷**(打包父行仅含缺失/不一致父);并断言剪枝与「不剪枝全量推送」应用后中心终态等价(UPSERT 幂等) | 一致父行打包数 = 0;剪枝前后中心终态校验和差异 = 0 |
| 一致性缓存只认中心权威态(C10) | 构造"B 推 P1 被高 rank D 仲裁掉"场景,断言 B 缓存**不得**记 `P1_B` 为中心态、后续引用 P1 的推送**不得剪 P1** | 误记中心态次数 = 0;该场景下 P1 打包数 = 1(必传) |
| 指纹抗碰撞/稳定/冷父必传(C11) | ① 同表两条不同行指纹不等;② 同一行重复算指纹恒等;③ 无中心背书的冷父打包数 = 闭包内该类父数 | 碰撞数 = 0;同行指纹方差 = 0;冷父漏传数 = 0 |
| 上行冲突算子分类(C12) | ① 依赖父在中心已存在但不同 → `DO NOTHING`,中心父值不被改写;② 直选记录 → 中心采纳 B 值并下行广播;③ `rank(D)>rank(B)` 但 B **直选**该行 → 中心采纳 `P_B`(人工选择优先于 rank) | 依赖父改写次数 = 0;直选采纳率 = 100%;并发推同 PK → `W_SYNC_CONCURRENT_MANUAL_PUSH` |
| 大推送分片/续传/超限(C13) | ① 超预算闭包 → 分片且"父片不晚于子片"、中途无 FK 报错;② 第 k 片后中断 → 续传 → 终态与不中断一致;③ 超硬上限 → `E_SYNC_SELECTION_TOO_LARGE` | 跨片 FK 报错 = 0;中断/不中断终态校验和差异 = 0 |
| 下行去抖攒批(C14) | ① 窗口内对中心做 N 次变更 → 每对端只产出 1 发合并广播;② 静默后广播数 = 0;③ 先触达阈值则即发 | N 变更→广播数 = 1/对端;静默广播数 = 0 |
| 前后台双面(C15) | ① 边缘长推送(Exporting 等 ACK)中,中心下行到达 → 边缘仍能应用并刷新一致性缓存;② 中心同时收 B、C inbound → 串行应用、均不报 `E_BUSY`;③ 前台 push 进行中再 `startImport` → `E_BUSY` | 后台被阻塞数 = 0;中心 inbound 拒收数 = 0;前台重入 → `E_BUSY` |
| 长推送护 WAL/漂移/续传(C16) | ① 长推送期间并发改选中行 → 不持长读事务(WAL checkpoint 正常、WAL 不无界增长),终态=最新内容且 FK 完整;② 推送期删选中行 → 剔除 + `W_SYNC_PUSH_ROW_DRIFTED`;③ 进程重启据冻结清单续传 | WAL 增长有界;漂移行告警数 = 漂移数;重启前后终态一致 |
| 推送撞迁移(C17) | ① 中心迁移中押旧版本片到达 → 整发 `E_SYNC_PUSH_SCHEMA_MOVED`、无半截脏态;② 新 schema 重新快照重推 → 正确收敛;③ 本地迁移某表 → 该表指纹缓存被作废(下次纳入不误剪) | 半截脏态记录数 = 0;重推终态正确;迁移后该表误剪数 = 0 |
| 载荷体积 | 单条增量载荷序列化后字节数不超过预算上限 | 单载荷字节 ≤ 预算〔TBD:设计阶段定 / 责任人待指派〕 |

## 10. 循序最小可落地（分阶段交付）

> 原则：每阶段一句目标 + 一句交付，下一阶段以上一阶段的可运行制品为前置。阶段 0 为可行性闸门，未过闸门不进入阶段 1。

| 阶段 | 目标 | 交付 |
|------|------|------|
| **阶段 0 预研（可行性闸门）** | 验证三大底层机制可用：启用 SQLite Session 扩展宏（重编译）、句柄穿透（在已打开连接上挂接 session）、rebaser 变基；并与第三方传输工具锁定 outbox/inbox 目录、文件命名、完成哨兵（sentinel）契约。 | 一份可行性结论 + 最小验证程序：能在目标编译产物上录制 changeset、对两路 changeset 跑通 rebaser 固化裁决、读写约定目录并识别完成哨兵；任一项不通过则触发降级/换路方案，**不进入阶段 1**。 |
| **阶段 1 两节点最小同步** | 打通单向→双向最小数据流：短命 session 捕获→`sync_changelog`→载荷打包→outbox/inbox 落地→`ChangesetApplier` 源优先回调应用→ACK→`applied-vector` 幂等。 | 两节点（A↔B）可双向同步增量；落地 `ISyncEngine` 8 接口骨架（FR-11 状态机最小实现）；重复投递的同一载荷经 applied-vector 去重不重复应用（FR-1/FR-3/FR-4/FR-5）。 |
| **阶段 2 星型广播与防回声 + 上行选择性推送** | 由两节点扩展到单域星型：`RoutingTable` + rebaser 固化裁决广播 + per-peer 锚点 + origin 优先级确定性仲裁；并落地上行人工选择性推送（FR-16/FR-17）。 | 星型拓扑下中心向各叶子广播且无回声；新增 `SelectionResolver`（人工选择记录）+ `FkClosureBuilder`（外键依赖闭包，拓扑序，UPSERT）+ `syncSelected` 接口；跑通 PNG 场景 1 闭环（FR-2/FR-6/FR-9/FR-16/FR-17）。 |
| **阶段 3 精简导入导出门面** | 在已验证的同步引擎之上提供门面层，不重写引擎。 | `IBatchTransfer` 非阻塞启动 + 轮询 + `stop`/`state`，复用现有引擎；单活动操作互斥（FR-15、对应 C9）。 |
| **阶段 4 场景 2 对比/合并 API** | 为宿主 GUI 提供库侧对比与合并能力，库不建 GUI。 | `DiffEngine` 表级红绿零拉取 + 行级只物化受影响行；`StagingBuffer` 的 `save`/`discard` + 会话隔离（陈旧检测 `E_SYNC_STAGE_STALE`）（FR-12/FR-13/FR-14，对应 C5）。 |
| **阶段 5 加固** | 覆盖异常路径与非功能指标，达到可交付质量。 | 基线/增量自动回退（FR-8）+ schema 版本隔离重放（FR-7）+ 死对端阈值逐出（FR-10）+ 故障注入 + 2Mbps 实测达标（验收数值见 §13 R5 待确认）。 |

## 11. 测试策略

> 分层覆盖：单元（纯逻辑）→ 模块（单组件含 SQLite 交互）→ 集成（多节点/端到端）→ 专项（确定性/防回声/冷启动/schema/死对端/场景2/门面/低带宽/故障注入）→ 回归。

| 层级 | 用例 |
|------|------|
| **单元** | PayloadCodec 打包/解包往返一致 + 指纹（fingerprint）校验；ConflictResolver 冲突→REPLACE/OMIT/ABORT 映射正确；RoutingTable 防回声判定（origin + 锚点）；DiffEngine 表级/行级判定（红/绿/差异行）；applied-vector 幂等去重（重复载荷不重复应用）；**FkClosureBuilder 外键依赖闭包递归展开 + 拓扑序正确（父先于子）** |
| **模块** | SessionRecorder 捕获完整性（增/删/改全覆盖，短命 session 边界）；ChangesetApplier 的 REPLACE / OMIT / ABORT 各分支；rebaser 变基（两路 changeset 固化裁决后输出一致） |
| **集成** | 两节点双向同步收敛；星型 B→A→{C,D} 广播到达；冲突场景源优先覆盖 C 本地值（FR-6）；**上行选择性推送：人工选记录 + 自动补全外键闭包 + UPSERT 到中心节点（FR-16/FR-17）** |
| **确定性** | 多源变更两种到达顺序（如 P1 先到 vs P2 先到）→ 收敛到同一终态（FR-6 规范序仲裁） |
| **防回声** | 系统静默（quiescence）后无新载荷产生（不自激循环）（FR-9） |
| **冷启动** | 基线回退（无增量起点→走基线）；增量缺口检测 `E_SYNC_GAP`（FR-8） |
| **schema** | 版本不匹配→隔离（quarantine）；迁移后重放隔离载荷成功（FR-7） |
| **死对端** | 永不 ACK 的对端达阈值被逐出（FR-10）；对端恢复后 outbox 截断 + 重建基线（rebaseline） |
| **场景2** | 表级红绿对齐；行级幽灵行（ghost row，仅一侧存在）对齐；`stage`→`save` 落盘；`stage`→`discard` 零落盘；并发陈旧 `E_SYNC_STAGE_STALE`（FR-12/FR-13/FR-14） |
| **门面** | `IBatchTransfer` 启动 + 轮询 `state`；并发触发 `E_BUSY` 互斥；与现有 DataBridge 同输入→同输出（一致性）（FR-15） |
| **低带宽** | 载荷字节计量（满足字节预算）；表级零拉取计数钩子（确认未发生全量拉取）（NFR-2Mbps） |
| **故障注入** | 载荷损坏 / schema 不匹配 / 外键破裂 / 半截文件（truncated）/ 第三方工具不可用 / 断连后恢复 / **上行悬挂父 `E_SYNC_FK_CLOSURE_MISSING` / 空选择 `E_SYNC_SELECTION_EMPTY`** |
| **回归** | 现有导入导出不破坏；Profile 全量同步路径不破坏（基于现有项目向后兼容） |

## 12. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| SQLite Session 扩展不可用（编译产物未启用） | 无法录制 changeset，方案根基不成立 | 阶段 0 可行性闸门先验证；以编译宏随库启用并固化到构建配置 |
| rebaser 可行性未知 | 多源固化裁决无法变基，收敛性无保证 | 阶段 0 将 rebaser 连同 Session 一起验证；不通过则停在闸门、改走替代收敛策略 |
| 句柄穿透在并发下被误用 | 跨线程操作同一连接句柄导致数据竞争/崩溃 | 句柄严禁跨线程使用；session 与其连接绑定单线程，接口契约中明确约束 |
| 第三方传输工具契约不明确 | 目录/命名/哨兵约定不一致导致投递丢失或半截读取 | 阶段 0 与工具方锁定 outbox/inbox 目录、文件命名、完成哨兵；契约写入文档 |
| 防回声逻辑出错 | 广播自激形成回声循环，载荷无限增长 | origin 标记 + per-peer 锚点双重判定；集成测试覆盖 quiescence 无新载荷 |
| 多写并发冲突 | 同一行被多源并发修改导致结果不确定 | 逻辑复制（changeset）+ 源优先级仲裁；禁用物理复制 |
| 2Mbps 链路下大载荷拥塞 | 同步延迟激增甚至积压 | 二进制载荷 + 压缩；超阈值告警并拆分批次（阈值见 §13 R5） |
| 死节点导致 outbox/磁盘撑爆 | 永不 ACK 的对端使积压无界增长 | 死对端阈值逐出 + outbox 有界（容量上限 + 截断） |
| schema 漂移导致永久脑裂 | 版本不一致载荷无法应用，节点状态分叉 | schema 版本号 + quarantine 隔离不匹配载荷 + 迁移后重放 |
| 场景 2 大表渲染卡死 | 宿主 GUI 拉全表导致卡顿/OOM | 库侧分页 + 行级只物化受影响行；宿主用 QTableView 虚拟化（库不建 GUI） |
| WAL 膨胀 | 长事务/大批量写使 WAL 文件持续增长 | 分批提交 + 手动 checkpoint |
| **上行外键依赖闭包过大 / 悬挂父** | 闭包膨胀拖垮单次推送；父记录缺失导致 UPSERT 失败 | 拓扑序生成（父先于子）+ 闭包前置校验；悬挂父报 `E_SYNC_FK_CLOSURE_MISSING`；一致性剪枝剔除已一致父行进一步压缩闭包体积与带宽 |
