# 可靠 UDP 分片传输（拆包 / 组包）设计文档 · 最小可落地版（MVP）

> 适用范围：`examples/sync-demo/udp_transport.{h,cpp}`（sync-demo 与 sync-suite 共用同一份传输层实现）
> 关联需求：生产环境带宽很小 → 每个 UDP 数据报 ≤ 800 字节 → 强制多分片 → 必须补齐分片重组的健壮性
> 本版原则：**最小可落地**。只做「全量重传 ARQ」——用最少机制修复三个健壮性缺口并跑通 800B。选择性重传（NACK）、发送节流、组件大拆分等均**不在本版范围**。

---

## 1. 背景与目标

### 1.1 为什么要做

当前 `UdpFileTransport` 的分片阈值 `kFragSize = 60000`，demo 工件（同步 changeset、约 1702B 的快照）几乎都小于它，运行时几乎恒为「单片直通」——分片/重组代码路径基本休眠。生产要求每个 UDP 数据报 ≤ **800B**，一旦压到 800B，任何工件都会被切成多片，随之暴露三个此前「存在但从未被触发」的健壮性缺口：

1. **无重传 / 无 ACK**：任一分片丢失 → 接收端 `complete()` 永不成立 → 半成品 `PendingMsg` 永久滞留（内存泄漏）→ 文件永远到不了 inbox → 上层卡到超时。片越小、片越多，多分片消息整体丢失概率越高。
2. **无 PendingMsg 超时 / 淘汰**：滞留半成品不被清理。
3. **msg_id 为随机 32bit**：两个并发文件的 msg_id 碰撞会把分片混进同一 `PendingMsg` 致数据损坏。

### 1.2 MVP 目标

用**最少的机制**把「拆包 + 组包」做到健壮且可测：

- **对外接口**：设置「每次传输最大字节数」（数据报总长上限）；
- **拆包**：文件 → N 个 ≤ maxBytes 的自描述数据报；
- **组包**：乱序 / 重复 / 丢失 / 越界 / 元数据不一致下正确重组，并**逐条修复三缺口**：
  - 丢包恢复 = **全量重传 ARQ**（发送端超时未收 ACK 就整条重发，直到 ACK 或放弃）；
  - 泄漏 = 接收端 pending **超时淘汰** + 发送端超 `maxRetries` **释放**；
  - 碰撞 = 归并键含**发送端点** + msg_id 改**单调计数器**。
- **验证**：sync-demo 与 sync-suite 把每次传输最大字节数设为 **800B**，端到端跑通。

## 2. 现状分析

### 2.1 现有拆包 / 组包

| 环节 | 现状（`udp_transport.cpp`） | 问题 |
|------|------|------|
| 拆包 | `sendFile()`：`readAll` → 按 `kFragSize` 切片 → 每片加 17B 头逐个 `writeDatagram` | 阈值是「净荷」非「数据报总长」；发完即不管，无重传 |
| 组包 | `handleDatagram()` + `PendingMsg`：按 `msg_id` 归并，`complete()` 判齐，`assemble()` 顺序拼接 | 无超时淘汰；键只用 `msg_id`；丢片即永久卡死；**不校验 `frag_idx` 越界与元数据一致性** |
| 归并键 | `QMap<quint32 msgId, PendingMsg>` | 跨发送端 `msgId` 可碰撞 |
| 并发 | `run()` 单线程内「收 → 发」交替；`stop_` 原子；`sentFiles_` 防重发 | 单线程 → ARQ 状态可**无锁**托管在 `run()` 内 |

现有 DATA 头 = 17 字节：`magic(4) | msg_id(4) | frag_idx(2) | frag_count(2) | total_size(4) | fname_len(1)`。

### 2.2 决定设计形态的三个关键约束

- **`run()` 单线程收发交替**：接收与发送在同一线程顺序执行 → **所有 ARQ 状态可为线程私有、无需加锁**。
- **配置须在 `start()` 前设置**：`maxTransmitBytes`、超时参数等设定后 `run()` 只读 → 无数据竞争。
- **同一 socket 收发**：socket 绑定 `localPort_` 并用它发送，故其数据报源端口即 `localPort_`；接收端得到的 `(sender, senderPort)` 就是发送端监听端点，**回发 ACK 天然可达**。

### 2.3 8 个构造点（改造时统一设 800B）

`examples/sync-demo/main.cpp`（中心1+边缘3）、`examples/sync-suite/Scenario1Runner.cpp`（中心1+边缘3）、`examples/sync-suite/Scenario2Model.cpp`（中心A/子B 各1）。

## 3. 设计原则（MVP）

1. **「最大字节数」= 数据报总长上限，不是净荷。** 净荷 = maxBytes − 头长 − 文件名长，且**逐文件计算**（文件名嵌在每片、长度可变）。简单把 `kFragSize` 设成 800 会得到 `≈850+` 的数据报，反而违约。
2. **先做全量重传 ARQ，不做 NACK。** 生产工件仅 KB 级，全量重传（超时未收 ACK 就整条重发）足够健壮且实现最简；选择性重传是后续优化，本版不做。
3. **RTO 兜底永不被关闭。** 只要一条消息未收到 ACK，发送端就按 RTO 周期性重发，直到 ACK 或 `maxRetries`——不引入任何「收到某种反馈后停止重传」的开关（那会在反馈丢失时永久卡死）。
4. **入站严格校验，宁丢勿污。** 对每个 DATA 片校验类型/大小/越界/元数据一致性/组装总长；任一不过即丢弃该片或淘汰该消息，绝不静默交付截断/损坏的数据。
5. **无锁 + 单调时钟。** ARQ 状态托管在 `run()` 单线程；用 `QElapsedTimer` 取单调毫秒作所有超时基准。
6. **归并键含发送端点。** 接收端 `pending` 键 = `(senderKey, msgId)`；发送端 `msgId` 改每 transport 原子单调计数器 → 碰撞在设计上根除。
7. **交付可观测。** 送达以 **ACK 为准**：首发改名 `.sending`，收 ACK 改 `.sent`，超 `maxRetries` 放弃改 `.failed` + 告警——上层可据此感知失败，而非静默丢失。

## 4. 设计方案（MVP）

### 4.1 分片与「最大字节数」

- 新增「数据报类型」字段后，**DATA 固定头 = 18B**：`magic(4) | type(1) | msg_id(4) | frag_idx(2) | frag_count(2) | total_size(4) | fname_len(1)`。
- 每片净荷 `M = maxTransmitBytes − 18 − fname_len`。
- 分片数用 64 位算：`fragCount64 = data.isEmpty() ? 1 : ceil(total / M)`；若 `fragCount64 > 65535`（`frag_count` 为 `quint16`）→ **分片失败**（不发送、文件改 `.failed`）。空文件特判 1 片（空体也要被传输并触发落盘）。
- **下限拒绝而非上夹**：`maxTransmitBytes` 若 `< 274`（`= 18 头 + 255 最长合法文件名 + 1 净荷`）则**拒绝设置**（保持默认、返回 false），不静默突破调用者给的上限。上限 65507。800 远高于 274，安全。
  - 例：文件名 `snapresp__<12>.payload`（约 30B），800B 下净荷 `M = 800 − 18 − 30 = 752`，1702B 快照 → `ceil(1702/752) = 3` 片。

### 4.2 线格式（大端；`type` 紧跟 `magic`，收端先校验 `magic + type` 再分派）

| 类型 | type | 布局 |
|------|:----:|------|
| **DATA** | `0x01` | `magic \| 0x01 \| msg_id \| frag_idx \| frag_count \| total_size \| fname_len \| fname[N] \| payload[M]` |
| **ACK**  | `0x02` | `magic \| 0x02 \| msg_id` |

- DATA 各分片的 `msg_id / frag_count / total_size / fname_len / fname` 完全相同，`frag_idx / payload` 各异——故接收端**不论哪片先到都能初始化元数据**。
- ACK 由接收端在「消息组齐落盘」后回发给该消息的发送端点。

### 4.3 MVP ARQ 协议（全量重传）

**所有消息（含空文件、单片）走同一路径**——单片消息也进 outbound、也要等 ACK 或 giveUp，避免语义分叉。

**发送端** —— `run()` 内的 `QHash<quint32 msgId, Outbound>`；`Outbound { destHost, destPort, msgId, datagrams[], filePathBase, lastSentAt, retries }`：
- **首发**：`fragmentMessage` 拆包 → 全部分片发出 → 文件改名 `.sending`（防 pollOutbox 重扫）→ 记入 outbound（`lastSentAt = now, retries = 0`）。
- **RTO 兜底**：`now − lastSentAt > RTO` 时——若 `retries < maxRetries`：**全量重发**所有分片、`retries++`、刷新 `lastSentAt`；否则：放弃，文件改名 `.failed` + `qWarning`（`E_UDP_DELIVERY_GIVEUP` 语义），删除 outbound。
- **收 ACK**：先校验 ACK 的 UDP 来源端点 == `outbound[msgId].dest`（不匹配则丢弃，防误包/旧包）；匹配则文件改名 `.sent`、删除 outbound。

**接收端** —— `FragmentReassembler`，`pending` 键 = `(senderKey, msgId)`，`PendingMsg { filename, totalSize, fragCount, frags{idx→payload}, createdAt, lastProgressAt }`；另有「已完成短表」`completed{(senderKey,msgId) → completedAt}`：
- **收 DATA**：先过 §4.4 校验；
  - 若 `(senderKey,msgId)` 在「已完成短表」→ ACK 丢失导致的重发 → **只回 ACK、不重复落盘**（幂等）；
  - 新消息 → 建 pending（记 `createdAt = lastProgressAt = now`）；已存在 → 校验元数据一致；
  - 存片：仅当 `frag_idx` 之前未收到过才刷新 `lastProgressAt`（**重复片不刷新进度**，避免推迟淘汰判断）；
  - `frags.size() == fragCount`（`complete()`）→ `assemble()` 并校验 `assembled.size() == totalSize`：通过则落 inbox（+ `.ready` 哨兵）→ **回 ACK** → 移入「已完成短表」；不通过则丢弃 pending + 告警（损坏）。
- **淘汰**：`now − lastProgressAt > reassemblyTimeout` 仍未完成 → 淘汰释放内存（覆盖「发送端已放弃」，防泄漏）。
- **短表过期**：`now − completedAt > completedRetention` → 清理，保持有界。

**典型时序（丢第 k 片）**
```
子B: DATA[0..n-1] ──(第 k 片丢)──► 中心A: 缺 idx=k，不 complete
子B: (RTO 到、未收 ACK) 全量重发 DATA[0..n-1] ──► 中心A: 收齐 → 校验 → 落 inbox → 回 ACK
中心A: ACK(msg) ────────────────────────────────► 子B: 改 .sent、删 outbound
```

### 4.4 入站严格校验规则（组包健壮性核心）

对每个 DATA 片，依次校验，任一不过即丢弃该片（元数据类不一致则可淘汰整条 pending）：
1. `dg.size() ≥ 18`；`magic` 正确；`type == 0x01`。
2. `fragCount > 0`；`fragIdx < fragCount`（**拒绝越界片**——否则越界片可把 `frags.size()` 凑满 `fragCount` 致 `assemble()` 静默截断）。
3. `dg.size() ≥ 18 + fname_len`（文件名长度不超实际报文）。
4. 该 `(senderKey,msgId)` 已有 pending 时：本片 `fragCount / totalSize / fname_len / fname` 必须与首片**逐字段一致**，否则丢弃该片（严重时淘汰 pending + 告警）。
5. 组齐后：`assemble().size() == totalSize`，否则判损坏、丢弃 pending。

### 4.5 三个缺口 → 对应处理方案（MVP）

| 缺口 | 处理方案 |
|------|---------|
| **① 无重传 / 丢片卡死** | **全量重传 ARQ**：发送端按 RTO 周期性整条重发，直到收 ACK 或 `maxRetries`；RTO **永不被任何反馈关闭**；放弃后改 `.failed` + 告警，上层可感知 |
| **② 无淘汰 / 内存泄漏** | 接收端 pending 记 `lastProgressAt`，`run()` 每轮 `evictStale()` 淘汰超 `reassemblyTimeout` 无进展者；发送端超 `maxRetries` 释放 outbound；「已完成短表」按 `completedRetention` 过期。三处共同保证内存有界 |
| **③ msg_id 碰撞** | 归并键改 `(senderHost, senderPort, msgId)`；`msgId` 改每 transport 原子单调计数器（从 1 递增）。两并发文件即便同 msgId 也因发送端点不同而隔离 |

### 4.6 参数与时间约束（自洽）

**约束层级**（必须满足，否则会误淘汰 / 重复交付）：
```
pollMs ≤ RTO
发送端寿命 = maxRetries × RTO
completedRetention ≥ 发送端寿命 + RTO        // 保证「giveUp 前的最后一次重发」仍能命中短表→再 ACK，不重复落盘
reassemblyTimeout  ≥ 2 × RTO                 // 至少能熬过一两次瞬时丢失再判淘汰
```

**默认值**（满足上述约束）：

| 参数 | 默认 | 说明 |
|------|------|------|
| `maxTransmitBytes` | 60000（**测试设 800**） | 单个 UDP 数据报总长上限 |
| `pollMs` | 50 ms | 收发轮询间隔（不变） |
| `RTO` | 500 ms | 无 ACK 时的整条重传间隔 |
| `maxRetries` | 5 | 放弃前重传上限（→ 发送端寿命 2500 ms） |
| `reassemblyTimeout` | 4000 ms | 接收端半成品无进展淘汰阈值（> 2×RTO） |
| `completedRetention` | 4000 ms | 已完成短表记忆时长（≥ 2500 + 500 = 3000） |

> 定时器基于 `waitForReadyRead(pollMs)` 轮询，属**软定时器**：实际触发有 `pollMs + 发送耗时` 级抖动，测试断言需容忍该抖动。

## 5. 设计实现（MVP）

### 5.1 组件（均在 `udp_transport.{h,cpp}` 内）

**① 拆包（纯函数，可单测）**
```cpp
struct FragmentResult { QList<QByteArray> datagrams; bool ok = false; QString error; };
// 把一条消息切成若干 ≤ maxTransmitBytes 的 DATA 数据报。
// 空 data → 1 个空净荷片(ok)。文件名 UTF-8 > 255 或 fragCount > 65535 → ok=false + error(供 pollOutbox 改 .failed)。
static FragmentResult fragmentMessage(quint32 msgId, const QString& filename,
                                      const QByteArray& data, int maxTransmitBytes);
```

**② 组包（有状态机，可脱离 socket 单测——组包健壮性的核心测试对象）** `class FragmentReassembler`
```cpp
struct RxEvent {
    enum Kind { None, Completed, NeedAck } kind = None;  // Completed: 落盘并回ACK; NeedAck: 重复消息只回ACK
    quint32 msgId = 0;
    QString filename;   // Completed 时有效
    QByteArray bytes;   // Completed 时有效（还原出的整文件）
};
RxEvent feed(const QString& senderKey, const QByteArray& datagram, qint64 nowMs); // 喂一个 DATA(含全部校验)
void evictStale(qint64 nowMs);   // 淘汰超时 pending + 过期「已完成短表」
int  pendingCount() const;       // 供测试断言「无泄漏」
```
§4.4 的全部校验、乱序 map 存片、`complete()`、`assemble()` + 总长校验、去重、短表，都在此类内。

**③ 发送在途**：MVP **不单独拆类**，直接在 `run()` 内用 `QHash<quint32, Outbound>` 承载（全量重传逻辑很简单）。

**④ `QThread::run()` 编排**（保持既有「收 → 发」骨架，插入维护步）
```
bind(localPort_); mkpath(inbox); QElapsedTimer clk; clk.start();
FragmentReassembler reasm(...); QHash<quint32,Outbound> outbound; QAtomicInt msgSeq(0);
while (!stop_.loadAcquire()):
    if sock.waitForReadyRead(pollMs):
        while sock.hasPendingDatagrams():
            读 dg + (sender,port); key = endpointKey(sender,port); now = clk.elapsed()
            switch parseType(dg):   // 先校验 magic
                DATA: ev = reasm.feed(key, dg, now)
                      if ev.kind==Completed: 写 inbox(ev.filename,ev.bytes)+.ready; 发 ACK(ev.msgId)→sender
                      if ev.kind==NeedAck  : 发 ACK(ev.msgId)→sender
                ACK : if outbound.contains(msgId) && outbound[msgId].dest==(sender,port):
                          改名 .sending→.sent; outbound.remove(msgId)
    pollOutbox(sock, outbound, clk, msgSeq):
        对每个就绪工件:
            r = fragmentMessage(++msgSeq, name, data, maxTransmitBytes_)
            if !r.ok: 改名 payload/ready → .failed; qWarning; continue
            逐片发往 dest(单/多 peer 路由不变); 改名 payload/ready → .sending
            outbound.insert(msgId, {dest, msgId, r.datagrams, base, lastSentAt=now, retries=0})
    for e in outbound:   // RTO 兜底
        if now - e.lastSentAt > RTO:
            if e.retries < maxRetries: 全量重发 e.datagrams→e.dest; e.retries++; e.lastSentAt=now
            else: 改名 .sending→.failed; qWarning; outbound.remove(e.msgId)
    reasm.evictStale(now)
```
- `pollOutbox` 的「`.ready` 就绪协议」「单/多 peer 目标解析 `extractTargetPeer`」**均不变**；仅把「切片 + `writeDatagram`」替换为 `fragmentMessage`，并把「发完改 `.sent`」改为「发完改 `.sending`、由 ACK/giveUp 决定终态」。
- `endpointKey(host,port)` = `host.toString()+":"+port`，仅作 map 查找键；发送端点的结构化信息（`QHostAddress`+port）另存于 `Outbound.dest`，**从不反解析该字符串**，故 IPv6 字符串形态不影响正确性（loopback IPv4 场景更无虞）。

### 5.2 交付状态机（文件名后缀）

`{name}.payload(.ready)` → 首发 → `.sending` → 收 ACK → `.sent`；或超 `maxRetries` → `.failed`（+ `qWarning`）。`.sending` 兼作 pollOutbox 去重（不再命中 `*.ready` 扫描），与既有 `sentFiles_` 双保险。

## 6. 设计测试（MVP）

### 6.1 单元测试（确定性、注入丢包；新增 `tests/unit/tst_udp_reassembly.{cpp,pro}`）

纯组件测试，**不依赖真实 socket / 网络**——直接调 `fragmentMessage` 产出数据报、**按需扣留若干**后喂给 `FragmentReassembler`，断言行为。丢包即「不喂某些片」，**对生产代码零侵入**（无需任何生产丢包钩子）。

| 用例 | 场景 | 断言 |
|------|------|------|
| `frag_sizes` | data size ∈ {0,1,M,M+1,多片} × 文件名长度 | 分片数正确；每报 ≤ maxBytes；头字段正确 |
| `roundtrip_inorder` | 顺序喂全部片 | `Completed`，还原 bytes 与原文逐字节一致 |
| `roundtrip_reorder` | 乱序 + 重复喂 | `Completed` 且一致（乱序/重复幂等，重复不刷进度） |
| `retransmit_recovery` | 先扣留第 k 片喂入 → 不 complete → 再把整条(或第 k 片)重喂 | 补喂后 `Completed`（模拟全量重传恢复） |
| `eviction` | 半成品静置超 `reassemblyTimeout` → `evictStale` | `pendingCount()==0`（无泄漏） |
| `collision` | 两个不同 `senderKey` 用相同 `msgId` 交替喂 | 各自独立 `Completed`，内容不串扰 |
| `dup_after_complete` | 完成后再喂同消息 DATA | 不重复交付；返回 `NeedAck`（触发再 ACK） |
| `reject_out_of_range` | `frag_idx >= frag_count` 的片 | 被丢弃；不污染 pending、不误判 complete |
| `reject_meta_mismatch` | 后续片 `fragCount/totalSize/fname` 与首片不一致 | 被丢弃 |
| `reject_size_mismatch` | 组齐但 `assemble().size() != totalSize` | 不交付、判损坏 |
| `frag_errors` | 文件名 > 255 / `fragCount > 65535` | `fragmentMessage` 返回 `ok=false + error` |

### 6.2 集成 / 端到端（两个 demo，`maxTransmitBytes = 800`）

- **sync-demo**：其验证路径设 800，校验四节点全域收敛；检查各节点 inbox，确认工件经**多分片**到达（`frag_count > 1`）、`.sent` 出现。
- **sync-suite**：`QT_QPA_PLATFORM=offscreen ./sync-suite --selftest` 设 800，校验场景1 全域收敛 + 场景2 比对 / 列级采用 / 写回；确认 1702B 快照被切成 **3 片**并正确重组、比对结论不变。

## 8. 风险 · 取舍（MVP）

- **线格式不兼容**：新增 `type` 字段与旧 17B 头不兼容。因 sync-demo / sync-suite 同源、同构建、一体升级，无跨版本包袱（运行中新旧二进制混用会互不识别，demo 场景不存在）。
- **默认 60000 行为微变**：旧 `kFragSize=60000` 是「净荷」，新 60000 是「数据报总长」，分片边界略有不同；但 demo 工件均 < 60000、实际仍单片，**无可观测差异**（文档如实说明，不宣称「完全不变」）。
- **全量重传的带宽代价**：高丢包时整条重发比选择性重传费带宽；但 demo 工件 KB 级、loopback 近无损，代价可接受。选择性重传（NACK）留作后续优化，本版不做。
- **背靠背发包**：800B 下大工件分片数增多，`run()` 背靠背 `writeDatagram` 可能压满内核发送缓冲；本版不做发送节流——依赖「送达以 ACK 为准 + RTO 全量重传」自愈（`writeDatagram` 失败的片会随下一次 RTO 重发）。发送节流留作后续优化。

## 10. 术语澄清（避免与「拆包组包」混淆）

- **`.ready` 两步就绪协议**（先写 `.payload` 再写 `.payload.ready`）：是「文件写完了 / 可安全读了」的**完成信号**，不是分片拆组。
- **`encodeSnapshots` / `PayloadCodec`**：是**序列化 / 编解码**（结构 ↔ 一个字节块），产出/消费「一个整块」，整块之后才交给传输层分片——不是网络分片。
- 真正的网络「拆包 / 组包」只在 `UdpFileTransport` 一处：本设计的 `fragmentMessage`（拆）与 `FragmentReassembler`（组）。

## 附录：本版对评审意见的取舍（最小可落地）

已纳入 MVP（Codex 评审 critical/high 修正）：数据报总长上限、拒绝下限而非上夹、`type` 字段、`fragmentMessage` 错误结果、`fragCount>65535` 守卫、`(端点,msgId)` 归并 + 单调 msgId、**入站严格校验（越界/元数据/总长）**、**RTO 永不被反馈关闭的全量重传**、`maxRetries` + 交付状态机(`.sending/.sent/.failed`)、ACK 绑定来源端点、pending 超时淘汰、已完成短表（时间约束自洽）、空/单片同路径、重复片不刷新进度。
